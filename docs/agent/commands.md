## 药园 Mini App 与内联菜单

用户在私聊主菜单点击 `药园` 后，如果 `GARDEN_MINI_APP_ENABLED=true` 且 `GARDEN_MINI_APP_URL` 已配置，Reply Keyboard 按钮只作为普通文本入口，Bot 应发送原 inline 文字交互版药园首页，并在同一消息的按钮区追加 `打开药园` WebApp 内联按钮；手动发送 `药园` 文本走同一入口。未启用 Mini App 时继续进入原 inline 多级菜单。

- 灵田管理：查看灵田、开垦灵田、选择空闲灵田种植、一键收获成熟药草。
- 药园成熟提醒必须保留文字交互按钮，例如 `前往灵田` callback，并在启用 Mini App 时于按钮下方追加 `打开药园` WebApp 按钮；成熟提醒不得只发送 Mini App 入口，避免移除原有文字交互路径。
- Mini App 必须从 Telegram 私聊里的 `打开药园` WebApp 内联按钮打开，直接复制 URL 到普通浏览器或 Telegram 外部浏览器不会携带 `initData`；前端应在 `initData` 为空时提示用户发送 `药园` 后点击 `打开药园` 重新打开，不应把这种情况伪装成后端读取失败。
- Mini App 顶部资源栏可常驻显示积分、已开灵田数和可收数量，底部提供灵田、种子、药草、药铺、丹方五个快捷入口；资源值和入口徽标只能来自已读取的后端状态，点击底部入口只能切换前端页面，不得直接触发资产动作。首次同步或读取失败时可展示农场式巡园/重试面板，重试只重新拉取 `/api/garden/state`。
- Mini App 灵田页可展示当前工具模式条、地块短动效、全局飘字和处理中遮罩，用于提示手势、播种、收获和后端确认中的状态；这些反馈不得携带价格、产量、收益、成熟时间、库存扣减或目标地块列表等结算字段，处理中遮罩也不得代替后端事务成功结果。
- Mini App 前端可以在成功读取 `/api/garden/state` 后用 `localStorage` 保存 5 分钟有效的 `garden_snapshot`，用于网络异常时展示离线园况、离线横幅和重连提示；离线园况只允许查看，不允许提交开垦、购买、种植、收获、回收、参悟丹方或炼丹等资产动作。前端请求可设置 8 秒超时、5xx/网络错误短重试和自动重连，页面隐藏时应暂停倒计时和定时同步，恢复可见后立即同步一次；这些机制只提升可用性和性能，不得绕过 Telegram WebApp `initData` 校验或后端事务。
- Mini App 写操作已完成但刷新园况失败时，应提示“操作已完成，园况刷新失败，正在重新同步”，随后进入只读重连态并只重新读取 `/api/garden/state`；不得自动重发开垦、购买、种植、收获、回收、参悟丹方或炼丹请求。
- Mini App 灵田倒计时应优先局部更新具体 DOM 文本、进度条 CSS 变量和成熟时间提示；只有作物阶段变化、成熟状态变化、用户切页或后端状态刷新时才重建对应结构。批量播种和一键收获可按地块播放阶梯动画，但动画中的地块列表只能来自当前已读状态，后端仍必须在事务内重新计算实际操作目标。
- Mini App 灵田网格可包在农场小院地图中，并作为灵田页内容首块展示，使用远山、木栅、草地、灵泉、透视田垄和轻量灵气强化经营场景感；田块必须规整两列对齐，不得通过错位 transform 造成整体偏移。小院可显示不遮挡地块的紧凑工具状态牌，但不得叠放会遮挡田号、状态、作物或挤压点击区域的管事气泡、收获篮、水井、仓棚等大型浮层；这些元素只能改变视觉层次，不得改变可开垦地块数量、开垦顺序、成熟状态、倒计时、地块选择规则或任何 API 入参。
- Mini App 顶部只保留紧凑资源栏，不得再放置第二套分类导航或大型主面板；灵田页内容首块必须是可操作的小院地图。田区固定展示 6 个田位并严格两列三行，任何通用空状态样式不得让空田跨列；已开垦地块、下一块可开垦荒地和后续锁定荒地同时可见。下一块荒地只有在积分足够时才调用开垦接口，积分不足或顺序未到的锁定地块点击只在状态栏提示原因。每块田显示田号、状态章、地块名和倒计时；地图下方依次使用巡园指引、收成熟/批量播/买种/药铺快捷入口、当前地块快捷条和手势/播种/收获工具栏，不再重复展开大型地块详情面板。所有快捷入口只能复用既有动作或页面切换，底部码头、页面切换和点击波纹必须尊重 reduced-motion，且不得作为资产成功依据。
- 种子商店：Mini App 使用货架和柜台式交互，展示账户可用积分、种子库存、成熟时间、预期产量和北京时间每日限购；货架支持全部、袋中、可买分段筛选，筛选只影响前端显示；筛选结果为空时需要提示当前筛选范围并提供回到全部的入口；货架可加入种袋、木板、柜台纹理、灵种铺招牌、浮尘、卡片底座等装饰层；货架卡片应标识已选、可买、售罄或缺积分状态，并可展示价格货签、种袋库存籽粒和限购余量条；购买仍调用后端种子购买接口，播种动作应回到灵田页执行。
- 草药背包：Mini App 使用仓库格子交互，仓库支持全部、有货分段筛选，筛选只影响前端显示；仓格可加入木箱、药秤、柜格纹理、仓格标签、卡片底座等装饰层；药草箱应标识已选、有货或空箱状态，并可展示库存强度条；点选药草只查看库存信息，回收信息放到药铺页。
- 药铺回收：Mini App 使用急收行情条和回收柜台交互，查看今日急收行情；急收行情条应显示当前库存可匹配数量、仅有库存或无库存状态，点击行情只选择对应药草；回收柜台可以预览当前输入数量的急收额度内数量、普通回收数量和预计积分，并可展示急收账台、算盘和额度进度条，但这些都只作为前端提示；回收 1 株或指定数量回收仍走后端回收接口，药铺页不承载炼丹动作。
- 文本回收：`回收灵草 玄参根 1`，只支持正整数数量，仅私聊执行；不要再接受 `全部` / `all` 或 `quantity=-1` 作为全量回收哨兵值。
- 丹方炼丹：Mini App 使用炼丹炉和卷轴式交互，炼丹炉需要有炉体、底火、火焰燃烧、蒸汽动态和符文环，也可以加入药柜、蒲团、卷轴堆、卷架纹理等炉房装饰；丹炉导引可按可炼丹方、缺料丹方、未悟卷轴或待命状态推荐下一步，按钮只能复用既有选丹方、切页或前端筛选入口，不得直接调用参悟或炼丹接口；丹方卷轴支持全部、可炼、未悟分段筛选，筛选只影响前端显示；丹方卷轴应标识可炼、缺料数量或参悟价格；点选丹方查看解锁状态、材料拥有量、炉火消耗和成丹库存；材料格可展示拥有/需求进度条；缺料丹方可显示寻药清单，点击缺料只能按草药名跳转并选中草药背包中的对应仓格；参悟丹方和使用已解锁丹方炼丹仍走后端事务接口。
- Mini App 前端物品图标使用 SVG 小图标而不是随机字符，灵田、种子、药草、药铺、丹方、收获、商店、倒计时等入口应保持稳定语义图标；凝露草、青灵叶、赤阳花、月华藤、玄参根、紫玉芝、龙血果和天心莲必须按真实物品 key 使用可辨认的专属造型与配色，不得仅随机换色或在本地 Mock 中使用生产配置外的名称。播种、收获、回收和炼丹可有轻量动效，但不得新增资产动作或后台状态。
- Mini App 灵田地块可按发芽、抽枝、将熟和成熟展示土壤呼吸、水汽、作物摆动、灵气粒子和采收冠，但每块田底部不得单独显示进度条；这些动态效果只能读取当前地块状态、成熟倒计时和工具模式，不得改变成熟判定、产量、库存、积分流水、API 入参或后端事务结果。
- 生长期作物必须按真实 `seedKey` 渲染对应灵草造型，不得让不同灵草长期共用同一嫩芽占位；阶段差异可通过缩放、透明度、摆动幅度、成长微光、聚灵阵和成熟冠表现。场景动画优先使用 transform 和 opacity，并必须受 `prefers-reduced-motion` 统一约束。
- 灵田可加入药庐、山门、日月、云层、灵泉、木桥、飞鸟和前景植被形成前中后景；种子铺、药仓、药铺和丹房可分别使用棚布、货架、种袋、药秤、箱笼、急收扫光、算盘、炉火、蒸汽、符纸和卷轴边缘动画。所有场景层必须 `pointer-events: none` 或位于操作目标下方，不得遮挡文字和按钮，也不得参与资产动作或状态判定。
- 丹药和丹方主视觉不得退化为仅显示名称首字的文字徽章。现有六种丹药应按丹方 key 使用稳定的无文字专属图形，丹方卷轴应通过药材节点、连线、炉心和对应核心纹样表达配方；名称仍以普通文本辅助识别。丹炉场景应有明确炉体结构、符阵、悬丹、炉火和药气层，但这些动画不得参与炼丹材料、炉火积分、成丹库存或成功状态计算。
- Mini App 在手机和 PC/Telegram Desktop 中应保持同一套手机画布比例；不要用桌面视口把药园主内容拆成双栏，也不要让灵田在桌面端变成不同列数。静态 CSS/JS 应带版本参数，服务端继续返回 no-store/no-cache 头，避免不同客户端拿到不同版本的图标和动效。
- Mini App 药园视觉以清晰、轻快的农场经营界面为主，辅以原创修仙药圃氛围；可使用远山、仙雾、木栅、灵泉、土壤田垄、灵种袋、药仓和丹炉等场景元素，但装饰不得遮挡田号、倒计时、价格、库存、材料数量或操作按钮，也不得影响点击命中、成熟判定、资产动作、API 入参或后端事务结果。
- 本地前端调试允许在 localhost URL 使用 `?mock=1`，即使 Telegram SDK 对象存在但 `initData` 为空也可加载 Mock 园况；该能力必须同时受 localhost 主机检查和显式 query 参数约束，禁止在生产域名绕过 Telegram WebApp 鉴权。

Mini App 调用 `/api/garden/*` JSON API，后端必须使用 Telegram WebApp `initData` 校验真实 Telegram 用户，不得相信前端传入的 user id。API 写操作仍复用 `gardenOpenNextPlot`、`gardenBuySeed`、`gardenPlantSeed`、`gardenHarvestPlot`、`gardenHarvestAll`、`gardenSellHerbQuantity`、`gardenBuyRecipe` 和 `gardenAlchemy` 等原事务函数。

作物成熟后会私聊发送一次成熟提醒，提醒中的 `前往灵田` 按钮复用 `garden:fields` 回调进入灵田管理；启用 Mini App 时同一提醒下方还会追加 `打开药园` WebApp 按钮。

群聊发送 `药园` 仅提示用户私聊 Bot，不在群内执行资产操作。

GitHub 福利入口为用户活动二级菜单 `GitHub福利`、私聊命令 `github福利` 和 `校验github福利 GitHub用户名`。群内触发只提示私聊 Bot，不执行 GitHub 校验、不创建卡密、不展示卡密。校验登录名只接受 GitHub 用户名、`@用户名` 或 `https://github.com/用户名` 形式，必须拒绝 `http://`、URL userinfo 和非 `github.com` 主机。校验方式固定为 GitHub Bio 放置每个 TG 用户随机生成的 1 小时校验码，满足注册满 5 年后按当前 TG 状态发奖：未绑定 ABS 账号发 1 个邀请码，已绑定 ABS 账号发 150 天续期卡；每个 TG 账号和每个 GitHub id 均只能成功领取一次。超级管理员查看 `查看github福利` 时，已领取和有效待校验计数读取失败必须显示“读取失败”，不得把查询错误当成 0。

宗门二级菜单包含 `宗门喇叭` 和 `世界喇叭` 入口。按钮只展示用法，实际发送必须私聊输入 `宗门喇叭 内容` 或 `世界喇叭 内容` 并完成二次确认。

宗门洞府命令包含 `洞府`、`解锁洞府`、`确认解锁洞府`、`闭关`、`确认闭关`、`宗门闭关`、`确认宗门闭关`。宗门 4 级开放洞府，成员消耗 100 个人贡献解锁；`贡献换声望` 同时增加宗门声望和个人声望。`贡献换积分` 已关闭，旧命令只提示关闭，不得产生积分或扣减贡献。个人闭关消耗 8 个人声望持续 2 小时；宗门闭关由宗主或长老消耗 `60 + 宗门等级 * 10` 宗门声望持续 4 小时。闭关期间只对闭关开始后的听书增量减缓天道压制，状态和确认文案必须明确消耗、持续时间、结束时间和当前余额。

世界 Boss 命令包含 `世界Boss`、`Boss状态`、`参加Boss`、`Boss排行`。世界 Boss 开启后最后 `15` 分钟停止新用户 `参加Boss`，且 `参加Boss` 必须校验用户拥有当前有效、未过期、未被管理员暂停、未处于播放异常临停的 ABS 听书账号。已成功参加者仍可继续造成伤害；但实时刷新和最终结算时若本地用户档案已不存在或已解绑 ABS，必须清零该参与记录的计算伤害，最终结算还必须从奖励排序中排除。`参加Boss` 成功后重读参与记录失败时，成功回执里的当前基线实际听书必须显示读取失败，不得误显示为 0 小时。Boss 击杀成功后生成 `30` 积分 / `10` 份 Boss 红包，结算公告需要提醒群内发送 `抢` 领取，但 Boss 红包仅限本期已成功 `参加Boss` 的修士；未参加者需要提示领取范围，不得把 Boss 限定红包误报为普通红包已抢空。`Boss排行`、实时战榜和结算公告读取排行失败时必须显示读取失败或稍后重试，不得把数据库异常当成暂无参与者。

宗门秘境支持普通、高阶和限时档位；每个宗门每周最多开启 2 次，普通、高阶和限时合并计算，按北京时间周一重置。状态、实时榜、结算公告、排行和明细需要展示秘境专属听书口径、档位、护道者、固定积分境界加成、贡献/声望倍率加成、积分/贡献/声望奖励和掉落。实时刷新和最终结算时若参与者本地用户档案已不存在或已解绑 ABS，必须清零该参与记录的计算奖励和掉落；最终结算还必须从护道者选择、奖励发放集合中排除。开启秘境后发送常驻实时榜并约每 2 分钟刷新，`宗门秘境排行` 会手动刷新当前 ABS 原始听书增量后展示榜单；秘境结束后 `宗门秘境` 状态查询应提示当前没有可参加的秘境，不再展示可进入指令。`进入宗门秘境` 成功后重读参与记录失败时，成功回执里的当前境界和基线净修为必须显示读取失败，不得误显示为凡人或 0 小时。`宗门秘境排行`、实时榜和结算公告读取排行失败时必须显示读取失败或稍后重试，不得把数据库异常当成暂无参与者。`宗门秘境明细 [用户ID]` 默认查看本人，查看他人仅限宗主、长老或超级管理员。

超级管理员秘境配置命令：

- 查看秘境配置

- 设置秘境倍率 档位 大境界 积分百分比 贡献百分比 声望百分比（积分百分比为历史兼容字段，当前积分按固定公式）
- 设置秘境掉落 档位 大境界 物品名 数量 概率百分比

秘境配置写入必须私聊执行、填写原因并二次确认，且写入审计日志。开启秘境、查看秘境配置和写入秘境配置读取配置失败时必须中止，不得按默认配置继续执行。

账号服务下的 `账户安全` 使用 inline 三级菜单展示，不再下发输入框下方的 ReplyKeyboard 子菜单。三级菜单包含：

- 修改密码
- 仅解绑不删号
- 删号注销

打开 `账户安全` 前必须显式读取本地用户档案；读取失败时提示稍后重试，不得当作未绑定账户。换绑安全码校验、删号注销安全码校验和仅解绑安全码校验也必须显式区分本地档案未找到与数据库错误；读取失败不得继续校验安全码、不得解绑、不得注销。

`删号注销` 为用户自助高危操作：必须先通过安全码校验并二次回复 `确认注销`；超级管理员账号禁止通过自助注销物理删除，必须先完成权限交接并移出超级管理员名单；确认注销时读取本地档案失败必须提示稍后重试且不得执行 ABS 删除或本地删除；ABS 删除失败时保留本地档案；ABS 删除成功或服务端已不存在后才允许物理删除本地档案。本地物理删除与 `SELF_DELETE_USER` 审计日志必须在同一数据库事务内完成；若本地删除或审计写入失败，需要提示用户立即联系管理员人工核查。

`修改密码` 必须先读取本地档案并校验安全码，再同步 ABS 新密码。本地档案读取失败时必须提示稍后重试，不得继续用空安全码校验或调用空 ABS ID 改密。

`绑定账号` 会创建或更新 Telegram 用户对本地资产档案的控制权。进入绑定、完成 ABS 密码校验后查询既有绑定、首次挂载或当前 Telegram 档案更新时，都必须区分本地档案未找到与数据库错误；读取失败必须提示稍后重试并记录脱敏日志，不得把数据库错误当成未绑定或首次接入继续流程。首次挂载或当前 Telegram 档案更新必须先完成 ABS 密码校验和安全码初始化，然后通过 `bindLocalUserWithAudit` 在同一事务内创建/更新本地 `User` 并写入 `BIND_USER` 审计；安全码换绑必须先完成 ABS 密码校验和原档案安全码校验，然后通过 `rebindLocalUserWithAudit` 在同一事务内按原始 `telegram_id` 和 ABS 用户 ID 条件更新 `telegram_id/status` 并写入 `REBIND_USER` 审计；`仅解绑不删号` 必须先通过当前档案安全码校验，然后通过 `unbindLocalUserWithAudit` 在同一事务内把 `telegram_id` 置为负数、状态置为 `unbound` 并写入 `UNBIND_USER` 审计。不得在状态机中直接创建/更新本地绑定档案后再单独写审计。

## 聚宝斋内联菜单

用户在私聊点击或发送 `聚宝斋` 后进入多级菜单：

- 丹药奇珍：按钮式选择商品并确认购买，货架正文、商品按钮、购买确认和购买成功提示都应展示系统丹药简要功效或短标识。
- 我的乾坤袋：查看当前 `Inventory` 中的物品；库存读取失败时必须提示稍后重试，不得把数据库异常当成空背包。
- 吞服丹药：跳转到既有乾坤袋吞服流程。

丹方、聚宝斋、乾坤袋和交易行中的丹药展示应包含简要功效说明：聚灵丹/九转造化丹展示手动吞服增加的丹药修为和周期限额，突破丹药展示对应突破用途。聚宝斋首页应直接显示丹药功效速览，方便用户在进入货架前确认用途。九转造化丹炼制消耗为炉火 `30` 积分、青灵叶 x8、玄参根 x3、紫玉芝 x1。吞服丹药事务成功后，若当前修仙档案读取失败，成功回执中的当前境界必须显示“读取失败”并记录脱敏日志，不得误显示为未知境界。

聚宝斋购买必须私聊执行。群聊只提示用户私聊 Bot，不在群内执行资产操作。

`注册账户` / `使用邀请码` 为用户私聊开户能力：进入注册前读取本地正式账号失败必须提示稍后重试并记录脱敏日志，不得把数据库错误当成未注册继续开户。邀请码输入后的预校验必须区分未找到与数据库错误；数据库错误提示稍后重试并记录脱敏日志，不得误报为邀请码无效。邀请码预占、退回和最终使用必须分别写入 `RESERVE_INVITE_CODE`、`RELEASE_INVITE_CODE`、`USE_INVITE_CODE` 审计，且不得保存邀请码明文。ABS 开户和 ABS 回滚属于外部副作用，不得放入数据库事务；本地档案写入失败且 ABS 回滚失败时，不得退回邀请码，需要提示用户联系管理员核查，避免同一邀请码继续制造遗留 ABS 账号。

`使用续期卡` 为用户私聊账号续期能力：卡密核销、到期时间延长和 `USE_RENEW_CODE` 审计必须在同一数据库事务内完成，审计不得保存续期卡明文。积分兑换生成的续期卡仅允许当前持有人本人使用，非持有人核销时必须给出明确提示并不得标记卡密已使用；该类卡密如需转让，只能由当前持有人通过交易行出售。试用用户使用普通续期卡时必须明确提示当前为新人体验账号、需先使用正式邀请码转正后才能使用，且不得核销卡密。若用户原本处于本地封禁状态，续期事务提交后再调用 ABS 恢复服务端账号状态；ABS 成功后，本地 `is_suspended=false` 与 `RENEW_REACTIVATE_USER` 审计必须在同一短事务内完成。ABS 或本地同步失败时，需要提示用户续期已到账但权限恢复需管理员核查，并写入对应失败审计；失败审计写入失败时必须通知超级管理员人工核查。

`积分兑换` 兑换邀请码或 30 天续期卡时，扣积分、写 `exchange_invite` / `exchange_renew` 流水和创建卡密必须在同一数据库事务内完成；事务成功后才允许发送明文卡密和“是否立即使用”的二次确认。二次确认会话只能保存卡密 HMAC 和脱敏预览，不得保存明文卡密。续期卡回复 `确认使用续期卡` 后复用普通续期核销逻辑，失败时必须提示卡密未消费。邀请码只有在用户是新人体验账号或未注册 ABS 账号时才提示直接使用；确认后，体验账号走正式邀请码转正流程，未注册 ABS 账号进入开户注册流程并预填该邀请码；已有正式 ABS 账号不得提示或直接消费邀请码。

## 卡密回收

私聊入口为 `资产交易 -> 卡密回收`，兼容 `卡密回收` / `回收卡密`。输入本服真实可用的邀请码或续期卡后查看当前配置价 60% 的报价，再回复 `确认回收卡密`。采用持有码规则，不要求原生成者或登记所有者；确认会话不得保存明文。
## 交易行

交易行入口位于用户主菜单 `资产交易` 的 inline 二级菜单，也兼容文字命令。`资产交易` 二级菜单直接提供 `交易行`、`我的购买`、`订单/举报帮助` 和 `上架商品` 入口；带参数命令继续通过私聊文字输入完成。

- `交易行` / `交易行列表`：查看最近在售、未到期且有库存的商品，并展示剩余有效期。
- `交易行帮助`：私聊查看订单号位置、买家举报、卖家查成交和管理员查单用法；管理员点击后台 `交易查单帮助` 也会进入同一说明。
- `交易行 卡密` / `交易行 背包` / `交易行 丹药` / `交易行 灵草` / `交易行 关键词`：按类型或关键词筛选在售商品；自定义关键词限 1-40 个字，不能包含换行、制表符或其他控制/分隔字符。
- `交易行详情 商品ID`：查看指定商品价格、库存、卖家、状态和剩余有效期；仅在商品仍在售、未到期且有库存时展示购买提示。
- `上架商品`：私聊选择 `1` 自由上架或 `2` 从背包上架；自由上架需达到筑基期及以上，分步输入商品名称、价格和逐行卡密，商品名称必须为 2-40 个字，且不得包含换行、制表符、其他控制字符或 Unicode 行/段分隔符；卡密单条最多 200 个字且不能包含制表符或控制字符；自由卡密仅允许 Bot 生成且尚未使用的邀请码或续期卡，未识别为系统卡密的内容不得上架，同一商品不得混放邀请码和续期卡；积分兑换续期卡只能由当前持有人本人上架；可识别原价值的自由卡密和背包物品单价都必须位于系统原价值的 85%-115%；背包上架分步输入物品名、数量和单价，物品名必须为 1-40 个字，且不得包含换行、制表符、其他控制字符或 Unicode 行/段分隔符。上架成功后会向公告群同步一条公开摘要，只包含商品 ID、类型、名称、单价、库存、来源标签、卖家展示名和查看/购买命令，不包含卡密明文。
- `我的交易行`：私聊查看自己上架的商品、库存、已售数量、剩余有效期和下架命令；商品售罄后会自动结束在售状态，不再等待 48 小时到期下架提醒。
- `我的购买` / `我的订单`：私聊查看最近购买记录；自由卡密可重新取回卡密，背包物品显示已交付乾坤袋。
- `购买商品 商品ID [数量]`：私聊购买；自由商品成交后 Bot 私聊发送卡密，背包物品成交后自动放入买家乾坤袋。数量不填默认为 1。
- `下架商品 商品ID`：卖家私聊下架自己的在售商品；背包物品未售库存会退回卖家乾坤袋。
- `交易行订单 商品ID`：卖家私聊查看自己某个商品最近成交订单。
- `举报订单 订单ID 原因`：买家私聊提交交易争议；原因限 3-200 个字，不能包含换行、制表符或控制字符。
- `查交易订单 订单ID`：管理员私聊只读查看交易订单和争议记录，不展示卡密明文；争议记录读取失败时必须明确显示读取失败，不得误显示为无争议记录。
- `强制下架商品 商品ID`：超级管理员私聊强制下架在售商品；必须输入合规原因并回复 `确认强制下架` 二次确认，关闭商品、关闭未售单位、退回未售背包库存和写入 `FORCE_CLOSE_MARKETPLACE_LISTING` 审计在同一事务内完成，事务提交后通知卖家。

带参数的交易行文字命令必须以空格或其他空白字符分隔命令名和参数，例如 `购买商品 12`；`购买商品12` 不会被交易行入口接管。

群聊只允许查看交易行列表和详情，以及接收上架成功后的公开摘要提醒；上架、购买、下架、购买记录和卡密交付都必须私聊执行。

交易行中的背包物品类丹药商品应在背包候选、公开列表、详情、购买确认、购买成功、卖家成交通知和上架群提醒中展示简要功效说明，文案来自统一的 `pillEffectSummary`；自由卡密商品即使名称与丹药相同，也不得展示丹药功效，避免伪装成系统丹药。自由卡密列表、详情、购买确认和购买成功应展示来源标签；Bot 产出的已校验邀请码/续期卡在最终购买事务内还要复核仍未使用，复核失败时交易中止且商品进入待核查；积分兑换续期卡成交时必须在同一购买事务内把卡密归属转给买家。

交易行商品状态为 `review` 时表示卖家数据异常待管理员核查；详情页不得展示购买命令，购买、手动下架和自动下架都不得继续执行资产流。

`天道奖池` 展示 `fusion_pool_points` 时必须显式处理数据库读取和配置解析错误；读取失败显示“读取失败”并记录脱敏日志，不得误显示为 0。

`财富榜` / `积分榜` 是私聊和群聊均可使用的只读入口；统计普通用户积分 Top 50，每页最多显示 30 人，通过 `wealth_page:` inline callback 按钮翻页，不支持文字页码翻页。榜单仍需排除本地 `role` 为 `super_admin` / `admin` 的用户以及环境配置中的管理员 Telegram ID。

`获取线路` 展示 `server_lines` 时必须显式区分未配置和读取失败；读取失败提示稍后再试并记录脱敏日志，不得误显示为未配置。展示正文必须使用 `serverLinesMarkdownBody`：先按 `validateServerLinesContent` 复核历史配置，再对正文执行 Markdown 转义；标题可保留 Markdown，线路正文不得直接拼接原始配置。

`邀请码价格` 和 `续期卡价格` 为超级管理员经济参数变更能力：必须输入新价格、调价原因并分别二次回复 `确认设置邀请码价格` 或 `确认设置续期卡价格`；价格范围为 0-100000 积分；读取当前价格失败或当前值不是合法整数时必须中止并提示稍后重试，不得用默认价继续确认；最终写入前必须再次校验价格范围；价格配置写入与 `SET_INVITE_PRICE` / `SET_RENEW_PRICE` 审计日志必须在同一数据库事务内完成；旧价格只能在事务提交成功后返回给调用方，失败时必须返回零值和错误。

`备份数据` 为超级管理员敏感数据外发能力：必须输入备份原因并二次回复 `确认备份`；执行前必须确认已配置 `BACKUP_GROUP_ID`；备份文件必须使用既有 AES-GCM 加密流程发送到备份群组；手动备份成功写入 `MANUAL_BACKUP` 审计日志，失败写入 `MANUAL_BACKUP_FAILED` 审计日志。成功或失败审计写入都必须显式检查结果，审计写入失败时通知超级管理员人工核查。

`清理遗孀` 为超级管理员 ABS 服务端遗留账号清理能力：必须私聊执行，先全量列出待删除 ABS 账号，输入清理原因并二次回复 `确认清理`；ABS 删除逐个执行，服务端已不存在按幂等成功处理。清理结束后写入 `CLEAN_WIDOWS` 审计日志；审计写入失败必须记录脱敏日志、通知超级管理员人工核查，并在最终回执中提示审计失败。

`备份状态` 为超级管理员只读运维能力：展示 `BACKUP_GROUP_ID` 配置状态、今日自动备份重试次数、最近成功日期/时间、最近尝试时间、最近备份消息、置顶消息、最近备份错误和置顶错误；该命令不得触发备份生成或发送，查看成功写入 `VIEW_BACKUP_STATUS` 审计日志。

`后台状态` 为超级管理员只读运维能力：展示后台调度窗口、生命周期巡检最近完成日期、生命周期最近错误、每日听书统计刷新结果、自动备份摘要和最近错误；该命令不得触发生命周期巡检、ABS 同步、自动备份或 Telegram 文件发送，查看成功写入 `VIEW_BACKGROUND_STATUS` 审计日志。

`审计概览` / `查审计概览 [天数]` 为超级管理员只读追溯概览能力：默认统计最近 1 天，天数只接受正整数且最多 30 天，展示审计总记录数、失败/异常记录数、高危操作数、Top 操作类型和 Top 操作者；该命令不展示审计 detail，不修改审计记录，查看成功写入 `VIEW_AUDIT_SUMMARY` 审计日志。

`审计日志` / `查审计 [过滤] [天数]` 为超级管理员只读追溯能力：默认查看最近 1 天，天数只接受正整数且最多 30 天，最多返回 20 条；过滤值为纯数字时匹配 `actor_id` 或 `target`，否则按 `action` 精确匹配。输出中的详情必须限长并按 Markdown 转义；查看动作本身写入 `VIEW_AUDIT_LOGS` 审计日志。

## 个人邀请拉新命令

- `我的邀请` / `邀请链接` / `拉新链接`：炼气初期及以上正式用户私聊生成或查看个人邀请链接，链接格式为 `/start ref_<code>` 或 `https://t.me/<bot>?start=ref_<code>`。
- `新人任务` / `检查新人任务`：试用用户私聊检查 7 天体验期内是否已累计听书满 10 小时；达成后领取 7 天新人体验延期。任务统计必须按试用开始到当前/结束的实际墙钟时长封顶，避免 ABS 日聚合把试用开始前的当日历史听书计入任务。
- `邀请统计`：正式用户私聊查看累计激活、有效新人和本月邀请奖励进度。
- 群聊触发上述命令只提示私聊 Bot，不在群内创建链接、检查任务或展示统计。
- 试用用户发送 `使用邀请码` 或注册/邀请码入口时，不重新开户，而是要求输入正式邀请码完成转正；转正后才能使用普通续期卡。
