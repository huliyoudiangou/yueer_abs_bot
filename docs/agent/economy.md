# Economy Rules

## 用户积分

用户积分字段：

- `User.Points`

所有用户积分变化都必须写入：

- `PointTransaction`

推荐使用：

- `applyPointDeltaInTx`

扣积分必须保证：

- `points >= 扣除金额`
- 不允许扣成负数
- 失败时事务回滚

数据库事务回调内不得调用 Telegram 发送/编辑消息、ABS API 请求或备份外发等外部副作用。资产状态应先在事务内提交，通知和外部服务调用放到事务提交后执行；外部动作失败时记录日志或通知管理员人工处理。

唯一索引冲突属于并发控制的一部分。生产代码判断数据库唯一约束错误时必须使用 `isUniqueConstraintError`，不要在各业务路径里直接用 `strings.Contains(err.Error(), "unique")`、`duplicate` 或 `constraint failed` 做零散匹配。

GORM 未找到记录错误必须使用 `errors.Is(err, gorm.ErrRecordNotFound)` 判断，不要直接 `err == gorm.ErrRecordNotFound`；这样被包装的查询错误仍能按业务语义正确处理。

后台任务或资产流程需要把最近错误写入 `SystemConfig` 时，必须通过统一 helper 脱敏并限长后再保存，并检查数据库 `Error` 与 `RowsAffected`；不要持久化原始 `err.Error()`，也不要把未实际写入的后台状态当成成功。

用户可见统计页不得把数据库读取失败当成 0。邀请统计、活动参与人数、宗门任务状态、Boss 参与和类似计数/聚合查询必须显式检查 `Count` / `Scan` / `First` 的错误；读取失败时展示“读取失败”或“状态暂不可用”，并记录脱敏日志。

## 兑换价格配置

邀请码兑换价格使用 `SystemConfig.invite_price`，默认 300 积分。

续期卡兑换价格使用 `SystemConfig.renew_price`，默认 150 积分。

用户兑换邀请码和续期卡时，扣积分必须在生成卡密的同一数据库事务内通过 `applyPointDeltaInTx` 完成，并分别写 `exchange_invite`、`exchange_renew` 流水；卡密生成失败时必须回滚扣分和流水。
用户兑换邀请码和续期卡时，事务内生成的明文卡密只能先保存在事务临时变量中；只有 `DB.Transaction` 返回成功后，才能发布到外层回执变量并发送给用户。事务失败或提交失败时不得让外层成功回执变量持有未提交卡密。
兑换成功后可以提示用户二次确认是否立即使用卡密，但该提示必须发生在扣分和卡密创建事务提交之后；确认会话只能保存卡密 HMAC 和脱敏预览，不得保存明文卡密。续期卡确认后必须复用普通续期卡核销逻辑，卡密核销失败时不得标记已使用。邀请码确认前必须重新读取当前账号状态：仅新人体验账号可直接转正，未注册 ABS 账号只能进入注册流程并预填邀请码，已有正式 ABS 账号不得提示或直接消费邀请码。
用户用积分兑换生成的续期卡必须写入 `RenewCode.Source=point_exchange` 和当前 `OwnerUserID`。该类续期卡仅允许当前持有人本人直接核销，或由当前持有人上架交易行出售；交易行成交时必须在购买事务内把 `OwnerUserID` 转给买家，避免买家收到卡密后无法核销，也避免私下转赠绕过交易行。

兑换商城展示价格、用户实际兑换扣费、超级管理员查看/二次确认价格时，必须显式处理 `SystemConfig.invite_price` / `SystemConfig.renew_price` 读取和整数解析错误。配置缺失或空值可使用默认价；数据库错误或非法整数值必须中止展示、扣费或调价流程并记录脱敏日志，不得静默使用默认价。

用户注册使用邀请码时，邀请码输入后的预校验必须区分未找到与数据库错误；数据库错误提示稍后重试并记录脱敏日志，不得误报为邀请码无效。邀请码预占必须通过条件更新把 `InviteCode.is_used=true`、`used_by_id=userID` 与 `RESERVE_INVITE_CODE` 审计放在同一数据库事务内；返回给调用方的邀请码记录只能在事务提交成功后发布，失败时必须返回空记录和错误。ABS 开户失败或本地档案写入失败且 ABS 回滚成功时，退回邀请码必须通过条件更新确认仍由该用户预占，并与 `RELEASE_INVITE_CODE` 审计放在同一事务内；如果本地档案写入失败且 ABS 回滚失败，不得退回邀请码，避免同一邀请码继续制造遗留 ABS 账号。注册首次创建本地 `User` 档案或复用已有档案写入新 ABS 账号时，都必须检查数据库错误和 `RowsAffected`；注册本地档案提交成功时，必须在同一事务内写 `USE_INVITE_CODE` 最终使用审计，审计只记录邀请码记录 ID 和脱敏 `code_preview`，不得记录邀请码明文。

管理员绑定 ABS 账号并创建本地 `User` 档案时，用户创建和 `BIND_USER` 审计必须在同一数据库事务内完成；创建本地绑定档案必须同时检查数据库错误和 `RowsAffected`，未实际写入必须回滚审计，避免绑定链路显示成功但本地档案缺失。

用户使用续期卡时，卡密核销、`User.expire_at` 延长和 `USE_RENEW_CODE` 审计必须在同一数据库事务内完成，积分兑换续期卡必须在标记使用前校验 `OwnerUserID` 与当前用户一致；审计 target 只能记录续期卡记录 ID，detail 只能使用脱敏 `code_preview`，不得记录续期卡明文。若用户原本处于本地封禁状态，事务提交后再调用 ABS 恢复服务端账号状态；ABS 成功后，本地 `User.is_suspended=false` 与 `RENEW_REACTIVATE_USER` 审计必须在同一个短事务内完成。ABS 恢复失败或本地状态/审计写入失败时，续期结果仍已到账，但必须写入 `RENEW_REACTIVATE_USER_FAILED` 或 `RENEW_REACTIVATE_USER_LOCAL_FAILED` 便于人工核查，且不得记录续期卡明文。

价格配置属于生产经济参数，必须满足：

- 仅超级管理员可执行；
- 新价格范围为 0-100000 积分；
- 必须输入原因并二次确认；
- 最终写入前必须重新校验价格范围；
- 写入 `SystemConfig` 时必须使用数据库级 upsert 并检查数据库错误，避免并发创建同一配置 key 时撞唯一索引；
- `SystemConfig` 写入和 `SET_INVITE_PRICE` / `SET_RENEW_PRICE` 审计日志必须在同一数据库事务内完成，审计旧值应在事务内重新读取且读取/解析失败时回滚，避免价格已生效但审计缺失或旧值不准。
- 调价函数返回给调用方的旧价格只能在事务提交成功后发布；事务失败或提交失败时必须返回零值和错误，避免后续回执或日志误用未提交事务中的旧值。

## 邀请码注册有效期

邀请码注册写入已有本地 `User` 档案时，`User.expire_at` 必须至少补到本次注册默认有效期 `now + ACCOUNT_VALID_DAYS`；若旧有效期更长，则保留旧有效期，不叠加、不缩短。`ACCOUNT_VALID_DAYS=0` 表示新注册为永久有效，已有有限到期时间的旧档案重新注册时也应清空 `expire_at`。

## 管理员卡密生成

超级管理员批量生成邀请码和续期卡属于资产型卡密创建。生成必须满足：

- 仅超级管理员可执行；
- 必须输入原因并二次确认；
- 邀请码数量范围 1-100；
- 续期卡天数范围 1-365，数量范围 1-100；
- 全部卡密记录创建和 `GENERATE_INVITE_CODES` / `GENERATE_RENEW_CODES` 审计日志必须在同一数据库事务内完成；
- 邀请码和续期卡共享创建入口必须检查数据库错误和 `RowsAffected`；未实际写入必须回滚当前事务，调用方不得把卡密明文发送出去；
- `InviteCode(code_hash)` 和 `RenewCode(code_hash)` 唯一索引必须只约束 `code_hash <> '' AND deleted_at IS NULL` 的有效行，启动迁移必须替换同名旧全量唯一索引或缺少 `deleted_at IS NULL` 的旧索引定义，避免软删除卡密历史阻塞新的卡密资产记录；
- 任一卡密创建失败或审计写入失败时整批回滚，不发送明文卡密；
- 返回给调用方的明文卡密列表只能在全部卡密记录创建和审计写入成功后发布，事务失败或提交失败时必须返回空列表和错误；
- 明文卡密只在事务提交成功后发送给授权管理员，不得进入日志或审计详情。

## GitHub 福利卡密发放

GitHub 福利会向通过校验的用户发放邀请码或续期卡，属于卡密资产流。必须满足：

- 用户入口仅私聊执行；群内只提示私聊，不创建校验码、不校验 GitHub、不发放卡密；
- GitHub API 请求属于外部副作用，不得放进数据库事务；
- 归属校验固定为 GitHub Bio 包含该 TG 用户本次随机校验码，校验码有效期 1 小时，账号注册时间必须满 5 年；
- 生成校验码只创建 `GithubBenefitClaim(status=pending)`，不扣减活动名额；
- 校验时发现 pending claim 已过期，需要先把该记录标记为 `expired`；状态更新失败必须提示稍后重试并记录脱敏日志，不得把数据库错误当成已成功过期；
- 最终发放必须在同一事务内复核 pending 未过期、活动已开启、TG 未领取、GitHub id 未领取、名额仍大于 0，随后按当前 TG 是否绑定 ABS 账号决定奖励：未绑定创建 `InviteCode`，已绑定创建 `RenewCode(days=150)`；
- 最终发放创建 `InviteCode` 或 `RenewCode` 时必须检查数据库错误和 `RowsAffected`；未实际写入必须回滚名额扣减、领取状态和审计，避免用户被标记已领取但卡密不存在；
- 名额扣减、卡密创建、claim 更新为 `claimed`、写入 `reward_type/reward_days` 和审计必须在同一事务内完成；邀请码写 `CLAIM_GITHUB_BENEFIT_INVITE`，续期卡写 `CLAIM_GITHUB_BENEFIT_RENEW`；
- 校验码 claim、剩余名额和最终卡密奖励等返回给调用方的事务结果，只能在事务提交成功后发布；事务失败或提交失败时必须返回零值和错误，避免发送未提交的卡密或展示回滚后的名额状态；
- 每个 TG 账号和每个 GitHub id 只能有一条成功领取记录，依赖 `github_benefit_claims` 上 claimed 状态的部分唯一索引兜底；启动迁移必须能替换同名旧全量唯一索引，确保 `telegram_id WHERE status='claimed' AND deleted_at IS NULL` 和 `github_id WHERE github_id > 0 AND status='claimed' AND deleted_at IS NULL` 两条约束都只作用于有效已领取记录；
- 卡密明文只能在事务提交成功后私聊发送给领取用户，不得写入数据库、日志或审计详情；审计只能记录卡密记录 ID、脱敏预览、GitHub id、奖励类型和必要上下文；
- 读取 `github_benefit_enabled` / `github_benefit_quota` 时必须区分配置缺失和数据库错误；配置缺失可按关闭或 0 名额处理，数据库错误或非法名额值必须中止领取、发奖或管理写入，不得误当成关闭/0 名额继续或写入不准审计；
- 管理命令 `设置github福利名额`、`开启github福利`、`关闭github福利` 必须仅限超级管理员私聊执行，输入原因、二次确认，并把 `SystemConfig` 写入和审计放在同一事务内；审计旧值必须在事务内用 checked 读取获取，失败时回滚；旧开启状态和旧名额只能在事务提交成功后返回给调用方。

## 管理员调账

人工调账使用流水类型 `admin_adjust`，并必须同时写 `ADJUST_POINTS` 审计日志。

调账必须满足：

- 仅超级管理员可执行；
- 可对任意已存在本地档案的 Telegram 用户调账，包括普通管理员、其他超级管理员和自己；
- 必须输入原因并二次确认；
- 最终确认执行前必须重新校验会话目标和调账数值；
- 单次绝对值最多 5000 积分；
- 每名超级管理员每日累计绝对值最多 20000 积分，每日窗口按北京时间 `[00:00, 次日00:00)` 计算；
- 入口预检和最终事务内复核都必须显式读取当日累计调账额；读取失败必须拒绝调账，不得当作 0 继续；
- 在数据库事务内通过 `applyPointDeltaInTx` 完成，扣减不得扣成负数。

## 背包

背包表：

- `Inventory`

唯一性：

- `user_id + item_name WHERE deleted_at IS NULL`

背包增加应使用 upsert，且 conflict target 必须匹配 `deleted_at IS NULL` 部分唯一索引，避免历史软删除库存行吸收新发货导致正常背包查询看不到到账物品。

背包扣除必须使用条件：

- `quantity > 0`

## 药园限购与丹方

药园每日种子限购、药草急收限额和丹方解锁都嵌入 `gorm.Model`，唯一索引必须只约束有效行：

- `garden_seed_purchases(user_id, seed_key, day_key) WHERE deleted_at IS NULL`
- `garden_herb_market_sales(user_id, seed_key, day_key) WHERE deleted_at IS NULL`
- `garden_recipe_unlocks(user_id, recipe_key) WHERE deleted_at IS NULL`

启动迁移不得只依赖 `CREATE UNIQUE INDEX IF NOT EXISTS`；同名旧全量唯一索引存在时必须先替换为部分唯一索引，避免历史软删除记录阻塞新的有效限购档案或丹方解锁记录。

灵田与生长中种植记录也嵌入 `gorm.Model`，唯一约束必须只约束有效行：

- `garden_plots(user_id, plot_no) WHERE deleted_at IS NULL`
- `garden_plantings(plot_id) WHERE deleted_at IS NULL AND status = 'growing'`

启动迁移必须检测并替换同名旧全量唯一索引，避免软删除灵田或已结束种植记录阻塞新的灵田初始化、开垦或播种事务。

## 丹药服用

手动吞服丹药必须在同一数据库事务内完成：

- 初始化或占用 `ItemUsageQuota` 额度；`ItemUsageQuota` 的唯一索引必须为 `user_id + item_name + period_key WHERE deleted_at IS NULL`，初始化 `ON CONFLICT DO NOTHING` 的 conflict target 必须匹配该部分唯一索引，并检查数据库错误和 `RowsAffected`，0 行只能表示本周期有效额度档案已存在；
- 通过 `quantity > 0` 条件扣减 `Inventory`；
- 写入 `ItemUsageLog`，并检查数据库错误和 `RowsAffected`；
- 累加 `Cultivation.pill_audio_time`。

任一步失败都必须回滚整笔事务，不得出现丹药已扣但修为药力未生效，或药力已生效但库存、额度、日志未同步的状态。

手动吞服确认前必须先读取当前周期 `ItemUsageLog` 计数；读取失败时应中止吞服并提示稍后重试，不能把失败当成 0 次继续确认。吞服事务成功后如果当前修仙档案读取失败，回执中的当前境界必须显示“读取失败”并记录脱敏日志，不得误显示为未知境界。`我的信息` 中体内丹毒沉淀展示同样必须区分读取失败，显示“读取失败/上限”，不得把查询错误当成 0。

后台修仙段位同步只能定向更新本函数负责的 `major_realm` / `minor_realm` 字段，并以读取时的 `major_realm` 作为乐观锁条件；更新必须检查数据库错误和 `RowsAffected`，未命中需记录脱敏日志，不得静默跳过。

修仙档案懒初始化依赖 `cultivations(user_id)` 唯一索引；创建时必须检查数据库 `Error` 与 `RowsAffected`，并发下 0 行写入表示档案已由其他请求创建，必须重读既有档案后继续，不得把唯一冲突或未写入静默伪装成可用新档案。

## 聚宝斋

聚宝斋入口：

- 一级菜单独立显示 `聚宝斋`。
- 私聊内使用 `shop:` inline 多级菜单。
- `积分兑换` 只保留邀请码和续期卡兑换；旧的回复 `3` 兼容跳转到聚宝斋。

聚宝斋购买必须：

- 在同一事务内扣除用户积分；
- 使用 `Inventory(user_id, item_name)` upsert 发货，物品名写入前必须先 `trim`；
- 写入 `shop_buy_item` 积分流水；
- 积分不足时不得发货；
- 发货失败时不得扣分。

按钮购买和旧数字购买都必须复用同一套购买事务，避免资产逻辑分叉。

聚宝斋首页应直接展示丹药功效速览；丹药奇珍货架正文、商品按钮、购买确认、购买成功提示和乾坤袋展示丹药时都应展示 `pillEffectSummary` 简要功效或短标识。功效文案仅用于展示，不参与价格、扣款、库存键或吞服/突破校验。

聚宝斋首页和丹药奇珍货架展示用户积分时，钱包读取失败必须显示“读取失败”并记录脱敏日志，不得把数据库错误当成 0 积分。

乾坤袋读取 `Inventory` 失败时必须明确提示稍后重试，不得把数据库错误当成空背包；否则会误导用户以为库存资产丢失或已被清空。

## 交易行

交易行相关表：

- `MarketplaceListing`
- `MarketplaceSecret`
- `MarketplacePurchase`
- `MarketplaceDispute`

交易行支持两种上架方式：

- 自由上架：用户自定义商品名称、商品价格和待自动分发的卡密；
- 从背包上架：用户从 `Inventory` 选择物品名和数量，系统锁定卖家库存，成交后自动转入买家乾坤袋。

自由卡密上架仅允许达到筑基期及以上的修士使用，判断口径为 `Cultivation.MajorRealm >= 2`。交互层选择自由上架时必须提前提示境界不足；`createMarketplaceSecretListing` helper 层也必须在事务内复核卖家境界，避免绕过上架向导直接创建自由卡密商品。未建立修仙档案的用户按凡人处理并拒绝。背包物品上架不受该境界门槛限制。

自由上架商品名称必须为 2-40 个字，且不得包含换行、制表符或其他控制/分隔字符；该限制必须在交互层和 `createMarketplaceSecretListing` helper 层同时校验。

上架、购买、下架和购买记录查询必须私聊执行；群聊只允许查看列表、详情和上架成功后的公开摘要提醒，不处理购买、卡密交付或背包转移。上架提醒必须在上架事务提交后发送，发送失败只记录日志，不得回滚已提交的上架；自由卡密上架提醒补读卡密来源失败时也只能记录脱敏日志，不得静默吞掉或回滚已提交的上架。提醒内容只允许包含商品 ID、类型、名称、单价、库存、卖家展示名和查看/购买命令，不得包含卡密明文、卡密预览或背包内部库存明细。

交易行商品有效期固定为 48 小时，`MarketplaceListing.ExpiresAt` 记录到期时间。新上架商品必须在创建事务内写入 `now + 48h`；历史 active 商品如果 `expires_at` 为空，Bot 启动成功并开始接收消息前只初始化一次为当时 `now + 48h`，后续重启不得重置已经写入的倒计时。

交易行购买、手动下架和自动到期下架前必须校验 `marketplace_listings.seller_id` 与同一商品全部 `marketplace_secrets.seller_id` 完全一致。若不一致，商品状态改为 `review` 暂停交易并通知管理员核查，不得继续成交、自动下架、给 listing seller 发到期通知或退回库存。

交易行错误分类必须使用稳定业务码；`marketplaceErrorCode` 对未知底层错误只能返回 `UNKNOWN`，不得把数据库、加密或外部错误字符串原样当作业务码传播。日志可继续记录原始错误用于排障。

列表查询支持：

- `交易行 卡密`：只看自由卡密商品；
- `交易行 背包`：只看背包物品商品；
- `交易行 丹药` / `交易行 灵草` / `交易行 种子`：只看对应关键词的背包物品；
- `交易行 关键词`：按商品名或背包物品名模糊搜索。

自定义筛选关键词必须为 1-40 个字，且不得包含换行、制表符或其他控制/分隔字符；超过限制时必须提示重输，不做静默截断或模糊替换。

公开交易行列表只展示可识别商品类型：自由卡密、背包物品和历史空类型自由卡密；异常 `listing_type` 不应进入公开列表。交易行列表、详情、购买确认和最终购买事务都必须排除已到期商品；交易行详情页如遇异常商品类型，不得提示购买命令，只能提示联系管理员核查。

交易行列表、详情、我的交易行和购买确认前的库存统计必须显式处理 `Count` 错误。读取失败时列表/详情/卖家页显示“读取失败”，详情页不得展示购买命令；购买确认前读取失败必须中止并提示库存状态暂不可用，不得把错误当成库存 0 或库存充足。

带参数的交易行文字命令必须以空格或其他空白字符分隔命令名和参数，例如 `交易行详情 12`、`购买商品 12`；命令名后直接拼接其他字符不应被交易行入口接管，避免误拦截普通聊天文本。

上架向导、购买确认、购买成功、卖家成交通知、背包候选列表和订单标题等用户可见商品名展示，必须经 `marketplaceVisibleItemName` 单行化和长度限制；Markdown 消息中的商品名必须经 `marketplaceMarkdownItemName` 先单行化再转义；交易行写入 `SellerName` 前必须经 `marketplaceStoredSellerName` 单行化、去控制字符和长度限制，公开列表和商品详情中的卖家昵称必须经 `displayMarketplaceSeller` 再次按同一规则兜底展示。该处理只影响交易行卖家昵称存储/展示，不改变真实商品名、背包物品名、卖家 ID 或卡密明文交付内容。

丹药类商品在交易行背包候选、公开列表、详情、购买确认、购买成功、卖家成交通知和上架群提醒中应展示 `pillEffectSummary` 简要功效；功效文案仅用于展示，不参与库存键、商品名、筛选或资产计算。交易行只有 `inventory` 背包物品类商品可以展示丹药功效，自由卡密商品不得仅凭名称展示丹药功效，避免卖家用商品名伪装系统丹药。

自由卡密商品需要展示来源标签：匹配未使用 `InviteCode` 的标为“系统已校验 · 邀请码”，匹配未使用 `RenewCode` 的标为“系统已校验 · 续期卡”。自由卡密仅允许 Bot 生成且尚未使用的邀请码或续期卡，未匹配 Bot 卡密表的输入必须拒绝上架；同一件自由卡密商品不得混放邀请码和续期卡，必须要求卖家拆分上架。历史三方来源标签仅用于兼容旧记录展示，不得允许新上架继续写入三方来源。背包上架商品已经由系统锁定库存，不额外增加卡密来源标签。

交易行新上架商品单价必须位于系统原价值的 `85%`-`115%`。自由卡密原价值按当前 `invite_price` 或按续期天数折算的 `renew_price` 计算；背包物品原价值优先使用聚宝斋售价、种子售价、药草基础回收价或炼丹材料成本价。无法识别原价值的特殊/历史背包物品不强制套用原价值区间，避免误伤非标准资产，但仍受全局价格上限限制。

卡密要求：

- 卡密明文只在卖家上架输入和买家成交/订单查询时出现；
- 数据库存储必须使用 `SECURITY_PEPPER` 派生 AES-GCM 密钥加密保存卡密；
- 同时保存 HMAC 和脱敏预览，列表和日志不得展示明文；
- 单次上架最多 50 条卡密，单条最多 200 个字，且不得包含制表符或其他控制字符；空行忽略、重复行去重，但任何非空卡密超过 200 个字或包含控制字符时必须整批拒绝并要求卖家重发，不能静默丢弃部分卡密；
- 自由卡密上架必须对本次输入去重，并禁止与现有 `available` 或 `sold` 卡密 HMAC 重复，避免重复交付或重复售卖；
- 自由卡密上架时必须只保存卡密 HMAC、密文、脱敏预览、来源和 Bot 卡记录 ID；不得记录卡密明文。匹配未使用 `InviteCode` / `RenewCode` 的 Bot 产出卡密，成交前必须在同一购买事务内按记录 ID、HMAC 和 `is_used=false` 再次复核；若为积分兑换续期卡，上架时必须校验卖家为当前持有人，成交时必须在同一购买事务内把持有人转给买家；复核失效或来源无法通过系统校验时，整笔交易必须中止，不得扣买家积分或给卖家入账，并应在事务回滚后用独立短事务关闭异常卡密、暂停商品为 `review` 待管理员核查。
- `marketplace_secrets(code_hash) WHERE code_hash <> '' AND status = 'available' AND deleted_at IS NULL` 部分唯一索引作为并发兜底；启动迁移如发现历史重复未售卡密，应保留最早一条 `available`，把其余重复未售卡密标记为 `closed` 并写系统审计，避免重复售卖同时不物理删除历史记录；已售 `sold` 卡密记录必须保留，供买家通过订单继续取回；该索引必须通过统一 helper 替换同名旧全量索引或谓词不完整索引，避免历史 sold/closed/空哈希记录扩大唯一约束；
- 买家可通过 `我的购买` 重新取回已购卡密，避免成交后 Telegram 私聊发送失败造成无法自助找回；取回前必须先校验订单交付类型，只允许自由卡密或历史空类型进入卡密读取路径，其他异常交付类型只提示联系管理员核查；取回时必须同时按订单的 `secret_id`、`listing_id`、买家 ID 和 `sold` 状态查询，避免异常历史订单引用错可售单位时泄漏卡密；卡密查询或解密失败必须记录订单、商品、卡密记录、买家和脱敏错误，禁止记录卡密明文或密文。

背包物品要求：

- 背包上架物品名必须为 1-40 个字，且不得包含换行、制表符或其他控制/分隔字符；该限制必须在交互层和 `createMarketplaceInventoryListing` helper 层同时校验，避免异常历史物品名进入交易行资产流；
- 上架时必须在事务内通过 `quantity >= 上架数量` 条件扣除卖家 `Inventory`，作为交易行锁定库存；
- 自由卡密上架和背包物品上架只有在商品主记录与全部可售单位都成功提交后，才能对外返回非零商品 ID；事务失败或提交失败时必须返回 0 和错误，避免回滚后的中间商品 ID 被通知、会话或后续逻辑误用。
- 每 1 个锁定物品对应一条可售单位，购买时通过条件更新抢占一条 `available` 单位；
- 购买成交后必须在同一事务内 upsert 增加买家 `Inventory`，物品名写入前必须先 `trim`，避免同名物品因空格分裂库存；
- 下架时必须在同一事务内统计未售 `available` 单位、退回卖家 `Inventory`，并把未售单位标记为 `closed`；关闭未售单位的 `RowsAffected` 必须等于本次退款依据的未售单位数，未命中或数量不一致时整笔回滚，避免退回库存后仍以可售单位留存；
- 下架事务只有在商品状态、未售单位状态和库存退回全部提交后，才能对外返回非零退款数量；事务失败或提交失败时必须返回 0 和错误。
- 超级管理员强制下架必须私聊执行，并要求原因和二次确认；商品关闭、未售单位关闭、未售背包库存退回和 `FORCE_CLOSE_MARKETPLACE_LISTING` 审计必须在同一事务内完成，事务提交后再通知卖家，通知失败不得回滚已提交资产状态。
- 已售购买记录保留，`我的购买` 只展示背包物品已交付，不展示卡密。

购买结算必须在同一数据库事务内完成：

- 商品必须处于 `active`，且未超过 `ExpiresAt`；
- 买家不能购买自己上架的商品，购买确认前和最终执行前都必须拦截自购；
- 支持 `购买商品 商品ID [数量]`，数量不填默认为 1，单次最多 50；
- 数量大于 1 或总价达到 1000 积分时，必须进入私聊二次确认，回复 `确认购买商品` 后才执行扣款和发货；
- 购买二次确认前和最终执行前都必须校验商品类型，只允许自由卡密和背包物品；历史空类型仅按自由卡密兼容，其他异常类型必须中止交易，不能按卡密路径兜底发货；
- 通过条件更新把指定数量的 `available` 可售单位标记为 `sold`，防止并发重复发放；库存不足时整笔交易回滚；
- 买家扣积分使用 `applyPointDeltaInTx`，流水类型 `marketplace_buy`；
- 交易行手续费为成交额的 5%，最低 1 积分，但不得吃掉单笔全部成交额；
- 卖家加积分使用 `applyPointDeltaInTx`，流水类型 `marketplace_sell`，金额为成交额扣除手续费后的实收；
- `marketplace_buy` / `marketplace_sell` 流水描述中的商品名必须使用单行化展示名，避免异常历史商品名用换行、制表符或控制字符打乱积分流水视图；
- 手续费在购买事务内注入天道奖池；锁顺序必须先持有 `fusionPoolMutex`，再开启数据库事务；
- 创建 `MarketplacePurchase`，`secret_id` 必须唯一，防止同一可售单位重复成交；
- 成交后必须在同一购买事务内重新统计剩余 `available` 可售单位；若库存已为 0，必须把商品主记录状态改为 `closed`，避免售罄商品继续保持 `active` 并在 48 小时到期巡检时误发下架通知；
- `marketplace_purchases(secret_id) WHERE secret_id > 0 AND deleted_at IS NULL` 部分唯一索引由一致性迁移显式创建；启动前必须检查历史重复购买记录，发现重复应禁止启动并人工核查；启动迁移必须能替换同名旧全量唯一索引，避免软删除或无卡密历史订单阻塞新的有效卡密订单绑定；
- 若并发下触发购买记录唯一约束，必须视为库存被抢先锁定，整笔交易回滚并提示用户刷新后重试；
- 任一步失败都必须回滚，不能出现扣分未发货或发货未扣分。

下架前必须校验商品类型，只允许自由卡密和背包物品；历史空类型仅按自由卡密兼容，其他异常类型必须中止下架，避免误退库存或误关闭可售单位。下架会改变商品状态为 `closed`，并把未售 `MarketplaceSecret` 单位标记为 `closed`。自由卡密未售部分不再进入购买流程；背包物品未售部分必须退回卖家乾坤袋；已售卡密和购买记录必须保留，供买家追溯。

交易行自动下架调度器每分钟扫描 `active` 且 `expires_at <= now` 的商品，必须复用与手动下架一致的事务路径完成关闭、未售单位关闭和背包库存退回；未售单位关闭行数必须和退款依据一致。若历史遗留商品已经没有 `available` 可售单位，巡检只静默关闭状态并记录日志，不得给卖家发送到期下架提醒。Telegram 私聊通知只能在事务提交后发送，发送失败只记录日志，不得回滚已提交的下架和库存退回。

订单与争议：

- 卖家可用 `交易行订单 商品ID` 查看本人商品最近成交订单；
- 管理员可用 `查交易订单 订单ID` 只读查看订单和争议记录；
- 管理员查询不得展示卡密明文，只能展示卡密脱敏预览；
- 管理员查询订单时，争议记录读取失败必须在订单详情中显示“读取失败”并记录脱敏日志，不得把数据库错误当成无争议记录；
- 订单列表、购买记录、争议记录和卡密预览等列表型展示字段必须经 `marketplaceDisplayText` 单行化和长度限制，避免用户输入换行、制表符、控制字符或 Unicode 行/段分隔符打乱核查视图；
- 商品类型、商品状态、订单状态、争议状态等枚举型展示字段不得原样回显未知历史值；`marketplaceTypeText`、`marketplaceStatusText`、`marketplacePurchaseStatusText` 和 `marketplaceDisputeStatusText` 必须用固定“未知类型/未知状态”兜底，避免异常数据库值进入 Markdown 或后台核查视图；
- 购买记录和订单行展示成交额时必须优先使用 `MarketplacePurchase.GrossAmount`；历史记录缺少成交额时，按 `Price * Quantity` 回退，缺失数量按 1 处理，异常负数成交额展示为 0；手续费展示必须钳制在 `0..成交额`；卖家实收展示必须优先使用 `MarketplacePurchase.SellerAmount`，缺失时按成交额扣手续费回退，异常历史记录不得展示负数实收，也不得展示超过成交额的实收；
- 买家可用 `举报订单 订单ID 原因` 提交争议，原因长度 3-200 字，且不得包含换行、制表符或其他控制字符；
- 同一买家对同一订单只能保留一个 `open` 状态争议，重复举报应提示等待管理员处理；
- 交易行争议通过应用层预查和 `marketplace_disputes(purchase_id, buyer_id) WHERE status = 'open' AND deleted_at IS NULL` 部分唯一索引双保险防重；启动前如发现重复 open 争议，必须禁止启动并要求人工核查；该索引必须通过统一 helper 替换同名旧全量索引或谓词不完整索引，避免已关闭或软删除争议阻塞新的 open 争议；
- 创建 `MarketplaceDispute` 必须检查数据库错误和 `RowsAffected`，唯一冲突继续映射为已有处理中争议，未实际写入不得提示提交成功；
- 若生产历史数据已有重复 `open` 争议，启动迁移不得自动清理；服务必须停止启动，管理员处理重复争议后重启即可补建索引；
- 提交争议前若无法确认是否已有 `open` 争议，必须中止创建并提示稍后重试，避免查询失败时重复制造争议；
- 争议只记录，不自动退款，避免被滥用；后续由管理员人工结合订单、流水和卡密预览处理。

## 积分盲盒

积分盲盒必须：

- 私聊执行；
- 开启前二次确认，并显示本次将扣除的积分、当前余额和扣除后余额；
- 用户回复 `确认开启盲盒` 后才扣除积分和生成奖品；
- 扣分和奖品生成必须在同一事务内完成，失败时不得扣分；
- 扣分必须写 `blind_box_cost` 积分流水；
- 奖品概率固定为：谢谢惠顾 71%、3 天续期卡 20%、专属邀请码 3%、30 天续期月卡 5%、365 天尊享年卡 1%；
- 确认后需要重新做余额条件更新，不能依赖确认前余额；
- 为未注册用户初始化幽灵钱包时必须检查 `User` 创建的数据库错误和 `RowsAffected`，唯一冲突只能重读既有档案，未实际写入不得继续扣分或生成奖品；返回给调用方的 `User` 和展示名只能在事务提交成功后发布，事务失败或提交失败时必须返回空档案和错误。

药铺回收：

- 基础回收按平均总回收额约为种子成本的 90% 计算；对当前可购买种子，按期望产量折算后的普通回收总额必须小幅低于种子成本，药铺普通回收只作为库存止损，不得成为稳定刷分来源。

- 炼丹产物进入 `Inventory`，MVP 产物对齐现有丹药：聚灵丹、筑基丹、降尘丹、九转造化丹、九曲灵参丹、补天丹。
- 丹方列表、聚宝斋货架/购买确认/购买成功提示和乾坤袋展示丹药时应展示 `pillEffectSummary` 简要功效；功效说明不得替代真实吞服额度和突破校验。
- 九转造化丹炼制成本为炉火 `30` 积分、青灵叶 x8、玄参根 x3、紫玉芝 x1，用于贴近聚宝斋 `350` 积分定价并保留种植耗时门槛。
- 丹方解锁依赖 `GardenRecipeUnlock(user_id, recipe_key)` 唯一索引。

## 红包

红包相关表：

- `RedPacket`
- `RedPacketGrab`
- `RedPacketLog`

`RedPacket` 的 `ref_type`、`ref_id`、`claim_scope` 用于表达系统活动红包来源和领取范围；字段只能通过 `AutoMigrate` 增量添加，不得清空历史红包。空 `claim_scope` 表示普通全员可抢红包。

同一红包同一用户只能抢一次，依赖唯一索引和事务。

红包创建时，`RedPacket` 记录、发包人扣分和 `redpacket_send` 流水必须处于同一数据库事务；扣分使用 `applyPointDeltaInTx`，任一步失败都必须回滚，不能出现扣分后无红包或有红包未扣分。
红包创建时，红包 ID 应作为事务内临时状态参与红包创建和 `redpacket_send` 流水写入；不得依赖事务外层红包 ID 变量承载中间结果，避免扣分、流水写入或提交失败后后续维护误用未提交红包编号。

制作积分红包输入金额后，钱包读取失败必须提示稍后重新输入金额并记录脱敏日志，不得把数据库错误当成 0 积分余额或余额不足，也不得清空当前创建会话。

红包领取必须防并发：

- 检查 `left_count`
- 检查 `left_points`
- 创建领取记录
- 更新红包剩余
- 增加用户积分
- 写积分流水
- 更新红包剩余必须使用 `id + left_count + left_points + is_finished=false` CAS 条件；并发抢占失败或领取唯一索引冲突时，必须回滚红包扣减和积分发放。
- 群内多人同时抢同一个红包时，CAS 失败和 SQLite 短暂写锁属于可重试冲突，应在外层重新开启短事务、重新选择当前最早可领取红包后再试；重试耗尽时提示用户重新发送关键词，不能在仍可能有剩余份数时直接宣称红包已抢空。
- 用户已经领取过当前所有活动中的同类红包时，应提示已领取过；只有不存在可领取红包时才提示已抢空或暂无红包。
- `claim_scope = world_boss_participant` 的红包仅允许 `ref_id` 对应 `WorldBossParticipant` 中存在参与记录的用户领取；不符合领取范围的红包不能被普通红包查询选中，也不能阻塞用户领取后续普通红包。
- 红包领取成功后的余额重读和抢空榜读取失败时，必须明确显示读取失败并记录脱敏日志，不得把数据库错误当成 0 积分或空榜；资产到账以领取事务和积分流水为准，展示失败不得回滚已提交领取。

## 赛马、骰子和牌九

赛马发起前必须先读取全服正积分用户的平均积分，用于计算本局下注范围；读取失败时必须中止发起并提示稍后重试，不得按 0 或默认平均值继续开赛。

赛马不得再注入系统补贴或庄家配资；最终奖池必须等于玩家总筹码 `userPool`，押中冠军者按本金比例瓜分玩家奖池，瓜分余数继续按既有规则进入天道奖池。该规则用于避免多号覆盖所有马匹后稳定套利系统增发积分。

牌九固定为每日 `18:00-19:55` 开放，骰子开放时间为 `22:05-次日17:55`，赛马保持 `20:00-22:00`；三个项目的入口分发必须避免 `押 3` 被赛马下注误吞。牌九下注固定为 `1-5` 积分，Bot 坐庄，使用 52 张扑克牌无大小王，A=1、2-9 按牌面、10/J/Q/K=0，两张相加取个位；玩家点数大于庄家时按 `本金 + 同额奖励` 发放，点数小于或等于庄家时下注归系统，同点文案为“同点庄家大半点”。庄家赢下的下注必须注入天道奖池。开局、状态和下注成功提示应展示完整牌规、剩余下注时间、参与人数和桌面总注；开奖结果必须明确参与群消息定时删除，且不置顶。

下注必须：

- 创建下注记录；
- 扣除积分；
- 写流水；
- 状态为 `active`。

异常中止必须自动退款。

赛马、骰子和牌九异常退款函数必须只在退款事务提交成功后返回退款人数和退款积分；事务失败或提交失败时必须返回 `0,0` 和错误，避免启动恢复、协程中止或日志误用未提交事务中的中间退款结果。

骰子每日净盈利上限依赖 `dice_daily_profits(user_id, day_key) WHERE deleted_at IS NULL` 部分唯一索引；增量 upsert 必须带同样的 `deleted_at IS NULL` conflict target，避免历史软删除行吸收更新后让正常查询看不到当日净盈利，导致每日盈利上限失真。

结算必须：

- 中奖发奖前必须先用 `status=active` 条件抢占对应下注行；
- 只有抢占成功的下注记录才能发放中奖积分、计入中奖名单和天道回收额；
- 系统通吃时如果本局存在下注池（`userPool > 0`），批量把 `active` 注单置为 `settled` 后必须检查 `RowsAffected`，未命中应回滚，避免奖池注入但注单状态未结算；
- 结算后更新下注状态；
- 写流水；
- 注入天道奖池时要安全。

牌九不做奖池瓜分、不做倍率；开奖前必须从数据库重读本局 `active` 下注作为结算快照，按下注创建时间发牌，并用 `status=active` 条件抢占每条下注行。赢家赔付、输家状态收尾、庄家赢额注入天道奖池和牌面结果写入必须在同一个事务内完成；注入天道奖池时必须通过 `runFusionPoolLockedTransaction` 先持有 `fusionPoolMutex` 再开启数据库事务，避免锁顺序反转。开奖群消息事务提交后使用自动删除发送方式发布，不置顶，发送失败只记录日志，不回滚结算。

## 天道奖池

奖池配置 key：

- `fusion_pool_points`

通过 `addPointsToFusionPool` 注入。

奖池水位写入依赖 `SystemConfig.fusion_pool_points`，初始化和更新必须收敛到统一 helper；初始化缺失配置时使用 `ON CONFLICT DO NOTHING` 并检查数据库 `Error` 与 `RowsAffected` 后重新读取，不得使用裸 `Create` 或先查再建作为并发兜底。`SystemConfig` 配置 key 唯一性必须由 `system_configs(key) WHERE deleted_at IS NULL` 部分唯一索引兜底，按 key upsert 的 conflict target 必须带同样的 `deleted_at IS NULL` 谓词，避免软删除历史配置或 SQLite 部分索引不匹配影响奖池状态写入。

事务内注入必须通过 `runFusionPoolLockedTransaction` 包裹后再调用 `addPointsToFusionPoolInTx`，确保锁顺序固定为先持有 `fusionPoolMutex`、再开启数据库事务。不要在普通 `DB.Transaction` 内直接调用 `addPointsToFusionPoolInTx`。

后台任务每日北京时间 12:00 后自动收集一次天道灵气，随机注入 5-10 积分到天道奖池。调度入口读取 `daily_fusion_pool_last_success_date` 失败时必须 fail closed：写入 `daily_fusion_pool_last_error`、记录脱敏日志、通知超级管理员并跳过本轮注入，不得把读取失败当成今日未收集。该任务通过 `SystemConfig.daily_fusion_pool_last_success_date` 在奖池事务内抢占当日执行权，抢占创建和后续状态 upsert 都必须检查 `RowsAffected`，再调用 `addPointsToFusionPoolInTx`，避免服务重启或多实例巡检造成重复注入；最近执行时间、注入数量和错误分别记录在 `daily_fusion_pool_last_success_at`、`daily_fusion_pool_last_amount`、`daily_fusion_pool_last_error`。注入事务提交后需要向 `NOTICE_GROUP_ID` 群提醒本次注入数量和当前水位，Telegram 发送失败只记录日志，不回滚已提交的奖池资产流。

达到 300 时生成系统红包。

## 世界 Boss

开放时间：
- 每周六、周日 21:00-22:00。

世界 Boss 伤害规则：

- Boss 血量按实际参与人数动态计算：基础 100，每名参与者 +12，最低 120，最高 450；
- 参加 Boss 时记录 ABS 原始总听书时长作为基线；
- 活动结束前最后 15 分钟禁止新增参加 Boss，避免临近结算新增参与者抬高血量导致无法击杀；
- 活动期间刷新或结算时再次读取 ABS 原始总听书时长；
- 基础伤害只按参与后的 Boss 时段实际听书增量计算：实际听书小时 × 30；
- 不计算白天净修为，不使用天道压制后的净修为；
- 实际听书增量必须按加入时间到当前刷新时间/活动结束时间的墙钟时长封顶，避免异常统计或多设备并发导致超过真实经过时间。
- 修为作为 Boss 伤害倍率加成：凡人 0%，炼气初期 +1%，之后每个小段位 +1%，最高 +25%；
- Boss 最终伤害 = 基础伤害 ×（1 + 修为加成 + 宗门科技加成），伤害保留小数点后两位；
- 世界 Boss 降临公告、参加回执、状态页、排行榜、实时战榜和结算公告中的 Boss 名称必须通过 Markdown 转义后展示，避免历史异常名称破坏消息格式；
- 取消单人 30 点伤害上限。

结算要求：

- 世界 Boss 启动前必须显式查询本期 `boss_id` 是否已存在；仅 `record not found` 才能创建新事件，数据库查询失败不得继续创建或发送降临通知，避免异常时重复公告或依赖唯一索引兜底；`world_boss_events(boss_id) WHERE deleted_at IS NULL` 唯一索引由启动迁移显式维护，必须能替换旧全量唯一索引；
- 宗门科技 Boss 伤害加成可以继续生效，并与修为加成加法叠加；实时刷新和结算读取修为或宗门科技加成失败时不得按 0 加成落库，实时刷新沿用旧伤害，结算回滚为 active 等待重试；
- 查询当前或最近世界 Boss 时必须区分“没有记录”和数据库读取失败；`参加Boss`、`Boss状态`、`Boss排行` 读取失败时必须提示稍后重试并记录脱敏日志，不得把数据库错误当成暂无 Boss；
- 活动进行中，`参加Boss` 成功后需要立即按当前参与人数刷新 Boss 血量；参与后血量、击杀状态和参与人数写回必须限定 active 状态并检查 `RowsAffected`，未命中不得当作刷新成功；`Boss状态`、`Boss排行` 和后台定时刷新需要实时重算参与者实际听书伤害、参与人数和 Boss 当前血量；
- `参加Boss` 成功后的参与记录重读失败时，成功回执里的当前基线实际听书必须显示“读取失败”并记录脱敏日志，不得把数据库错误当成 0 小时；已提交的参加记录不得因此回滚；
- 活动开始后发送一条常驻实时战榜；后台每 2 分钟编辑同一条消息刷新，不额外刷屏；榜单排序按最终两位小数伤害、实听时长、加入时间依次比较；
- `Boss排行`、实时战榜和结算公告读取排行失败时必须明确显示“读取失败”或提示稍后重试，并记录脱敏日志；不得把数据库错误当成暂无参与者；
- 活动进行中如果刷新后 Boss 当前血量归零，应立即进入幂等结算，不等待 22:00；
- 参与记录依赖 `world_boss_participants(boss_id, user_id) WHERE deleted_at IS NULL` 部分唯一索引，启动迁移必须能替换同名旧全量唯一索引；SQLite upsert 必须使用不指定冲突列的 `ON CONFLICT DO NOTHING`，并检查数据库错误和 `RowsAffected`；0 行写入只能按重复参加的幂等结果处理，不得覆盖首次参加基线；
- 伤害低于 `worldBossValidDamage` 的参与记录不发放奖励。
- 结算发放奖励后，参与者 `is_rewarded` 幂等标记、宗门贡献和宗门声望更新都必须检查 `RowsAffected`；未命中应回滚，避免写出 Boss 奖励流水但资产或幂等状态未到账；
- 结算事务成功后重读 Boss 事件用于刷新实时战榜；重读失败必须记录脱敏日志，不得静默吞掉，也不得回滚已提交奖励，结算公告必须使用本次结算已知的 `settled` 状态、剩余血量、击杀状态和参与人数补齐事件快照后发送，避免旧 `settling` 快照误导用户；
- 击杀成功时创建 `30` 积分 / `10` 份 Boss 红包，红包 `ref_type=world_boss`、`ref_id=boss_id`、`claim_scope=world_boss_participant`，仅本期参加 Boss 的用户可抢；结算公告需要提醒发送 `抢` 领取；普通红包和天道机缘红包不得受该限制影响；
- 击杀成功时天道奖池 `+10` 必须与结算同事务提交；有效参与者积分 `+1`、宗门贡献 `+3`，前三名和 Top 10 额外奖励保持既有排行规则；
- 未击杀时不发积分、不发 Boss 红包、不注入天道奖池；有效参与者宗门贡献 `+1`，前三名额外贡献 `+3/+2/+1`。

## 积分抽奖

抽奖相关表：

- `LotteryActivity`
- `LotteryPrize`
- `LotteryParticipant`
- `LotteryWinner`
- `LotteryClaimLog`

第一版支持两种开奖模式：

- 满指定人数自动开奖；
- 到指定时间自动开奖。

定时开奖创建时支持：

- 分钟数，例如 `60`；
- 固定时间，例如 `2026-06-07 20:00:00` 或 `2026-06-07 20:00`；
- 固定时间按北京时间解析，不依赖服务器本地时区；
- 固定时间必须晚于当前时间。

参与和开奖要求：

- 同一活动同一用户只能参与一次，依赖 `lottery_participants(activity_id, user_id) WHERE deleted_at IS NULL` 部分唯一索引，启动迁移必须能替换同名旧全量唯一索引；
- 创建抽奖时，`LotteryActivity`、全部 `LotteryPrize` 和 `CREATE_LOTTERY` 审计必须在同一事务内提交，审计失败时不得留下已创建活动；
- 抽奖创建后需要向群内发布介绍消息，介绍消息不自动删除，并尝试置顶；
- 活动名称创建时必须为 2-60 个字，且不得包含换行、制表符或其他控制/分隔字符；标题会进入群公告、活动列表、积分流水和审计上下文，不能静默改写；
- 抽奖列表、详情、创建公告、开奖结果、中奖私聊提醒和领奖成功提示中的活动名、中奖用户名和奖品展示名必须经 `lotteryDisplayText` 单行化和长度限制；未知历史状态必须显示为“未知状态”，不得原样回显异常状态值；
- 抽奖活动列表和详情页展示参与人数时，读取失败必须明确提示“读取失败”，不得把数据库错误当成 0 人；
- 抽奖详情页读取奖品或中奖记录失败时，必须明确提示读取失败并中止不完整展示，不得把数据库错误当成空奖品或无中奖记录；
- 奖品配置支持 `积分 数量 份数`、`续期 天数 份数`、`邀请码 份数`、`丹药 名称 份数`、`人工 奖品名称 份数`、`卡密 奖品名称|卡密内容`；人工奖品名称可包含空格，自定义卡密每行代表一份且单个活动总奖品仍最多 100 份；
- 丹药名称必须是系统已知丹药，例如 `聚灵丹`、`九转造化丹`、`万年仙玉髓`、`引灵入体`、`筑基丹`、`降尘丹`、`九曲灵参丹`、`补天丹`；
- 活动可配置参与消耗积分，0 表示免费；
- 付费抽奖参与时，创建参与记录和扣除参与积分必须在同一事务内完成；
- 限定人数抽奖参与时，创建参与记录后必须在同一事务内二次复核人数，超过上限要回滚参与和扣分；
- 参与扣分必须写 `lottery_entry_cost` 流水；
- `lottery_entry_cost`、`lottery_reward` 和 `lottery_cancel_refund` 积分流水描述中的活动名称必须使用 `lotteryPointDescriptionTitle` 单行化，避免异常历史标题打乱流水视图；
- 用户可以在群内发送 `参加抽奖 活动ID` 参与，成功后群内回复该用户原始消息并反馈当前参与人数；首次从群内参与需要在参与事务内回填活动公告群 ID，回填必须限定 active 状态并检查 `RowsAffected`，若并发中已由其他参与事务写入公告群 ID 可以继续，否则应回滚参与事务；
- 抽奖参与事务为本地无档案用户补建 `User` 时必须检查数据库 `Error` 与 `RowsAffected`；并发下 0 行写入只表示本地档案已存在，后续参与记录创建和付费扣分仍必须继续使用事务内条件写入兜底。
- 抽奖参与事务只有在参与记录、付费扣分和人数复核全部成功提交后，才能对外返回非零参与人数和满人开奖标记；事务失败或提交失败时必须返回 `0,false` 和错误，避免回滚后的中间人数或开奖标记触发错误回执/自动开奖。
- 开奖通过 `active -> drawing -> drawn` 状态流转防止重复开奖；
- 超级管理员手动 `强制开奖` 时，中奖记录生成、活动结果写回、天道奖池注入和 `FORCE_DRAW_LOTTERY` 审计必须在同一开奖事务内完成，审计失败时不得留下已强制开奖结果；
- 同一活动同一用户最多中奖一次，依赖 `lottery_winners(activity_id, user_id) WHERE deleted_at IS NULL` 部分唯一索引，启动迁移必须能替换同名旧全量唯一索引；
- 未中奖者不返还参与积分，避免参与扣费、返还和天道奖池注入叠加导致积分膨胀；
- 本期总参与积分的 10% 在开奖事务内注入天道奖池，向下取整；
- 抽奖开奖注入天道奖池时，锁顺序必须是先持有 `fusionPoolMutex`、再开启数据库事务，避免与其他奖池注入路径死锁；
- 活动取消时必须全额退还已参与用户，并写 `lottery_cancel_refund` 流水；
- 活动取消退款时，单个参与者必须先通过 `id + activity_id + is_refunded=false` 条件更新抢占退款标记并检查 `RowsAffected`，再发还积分；活动累计退款字段也必须检查行数；活动关闭、参与者退款标记、积分退还、累计退款额更新和 `CANCEL_LOTTERY` 审计必须在同一事务内完成，审计失败时整笔取消回滚；返回给命令层的累计退款额只能在事务成功提交后发布，事务失败或提交失败时不得使用事务内中间累计值；
- 开奖后优先在实际参与群公告中奖者，未记录参与群时回退到 `NOTICE_GROUP_ID`；
- 开奖结果公告不自动删除，并尝试置顶；
- 领奖暗号创建时必须为 3-40 个字，且不得包含换行、制表符或其他控制/分隔字符；
- 领奖暗号保存 HMAC 用于校验，并加密保存用于开奖后私聊提醒，不保存明文。

领奖要求：

- 用户必须私聊 Bot 发送领奖暗号；
- Bot 开奖后会私聊中奖者发送领奖暗号；
- 暗号只触发查询，奖品归属必须以 `LotteryWinner` 为准；
- 领奖期限固定为开奖后 2 小时，过期未领则不发奖，并提示用户奖品已作废；
- 只有 `pending` 中奖记录可以被过期任务或过期领奖尝试标记为 `expired`；已领取的 `claimed` 记录不得因截止时间已过被改写或提示为作废；
- 领奖发奖事务内必须再次复核截止时间；如果进入事务后才发现已过期，必须提交 `expired` 状态和 `LotteryClaimLog` 后返回 `LOTTERY_CLAIM_EXPIRED`，不得继续发奖，也不得把已提交过期状态回滚成待领取；
- 兑奖截止后，后台任务需要取消抽奖介绍和开奖结果公告的置顶；完成标记 `pins_unpinned` 必须限定当前状态、兑奖截止时间和未清理状态并检查 `RowsAffected`，未命中需记录脱敏日志，不得静默吞掉；
- 明确发送 `抽奖领奖 暗号` 且暗号错误 5 次后，使用 `SecurityAttemptLock` 暂停受理该用户消息 30 分钟；锁定剩余时间展示按真实剩余时长向上取整，锁定期内重复失败不得延长既有锁定窗口；
- `SecurityAttemptLock(user_id, purpose)` 唯一索引必须只约束 `deleted_at IS NULL` 的有效行；已执行旧安全锁迁移的生产库必须通过补迁移替换同名旧全量唯一索引，避免历史软删除锁记录阻塞新的失败次数记录；
- 积分奖品必须使用 `applyPointDeltaInTx` 发放，并写 `lottery_reward` 流水；
- 邀请码和续期卡在领奖事务内生成，使用现有 HMAC 卡密存储逻辑；
- 丹药奖品必须在领奖事务内写入 `Inventory`，同一用户同一物品使用 upsert 累加库存；`人工` 奖品在暗号领取事务成功后只返回“联系管理员领奖”提示，不引入待发放/已发放状态或管理员确认命令；
- 丹药领奖读取 `LotteryPrize.DisplayName` 时必须同时校验 `prize_id`、`activity_id` 和 `prize_type` 均匹配当前 `LotteryWinner`，不得仅凭奖品 ID 跨活动读取展示名或发货物品；
- 中奖记录领取时必须先通过 `id + activity_id + user_id + status=pending` 条件更新抢占 `claimed` 状态并检查 `RowsAffected`，再发放积分、邀请码、续期卡或丹药，防止重复领奖；
- 邀请码、续期卡和自定义卡密的 `prize_code_preview` 必须在同一事务内单独条件更新并检查 `RowsAffected`，不得使用整行 `Save` 覆盖中奖记录。自定义卡密必须逐份写入 `LotteryPrize.SecretEncrypted`，使用独立 AES-GCM 域密钥加密，数据库、审计和日志不得保存或输出明文；领奖事务必须按 `prize_id + activity_id + prize_type` 复核奖品后解密并私发，解密或发送失败沿用既有领奖失败告警边界。

## 修仙突破

修仙默认配置种子创建 `CultivationRealmConfig`、`CultivationMinorRealmConfig` 和 `BreakthroughConfig` 时必须检查数据库 `Error` 与 `RowsAffected`；唯一冲突或 0 行写入只能按默认配置已存在处理，不得在未实际写入时静默继续造成配置半初始化。启动迁移必须把大境界、小境界和突破配置的旧全量唯一索引替换为 `deleted_at IS NULL` 部分唯一索引，并在索引修复后补跑默认配置种子，避免旧索引曾经挡住缺失默认配置重建。

突破资源可能是：

- 背包丹药；
- 自动代购积分；
- 免费引灵入体。

突破预检必须先成功读取或初始化修仙档案；读取失败时提示稍后重试并中止，不得继续读取冷却期、境界、总修为或进入自动代购。突破检查自动代购前必须显式读取钱包和突破丹库存；读取失败时提示稍后重试并记录脱敏日志，不得把数据库错误当成 0 积分、余额不足或无丹药后继续代购。

突破必须记录：

- `BreakthroughAttempt`

成功或失败都应有记录，方便争议和排障。
`BreakthroughAttempt` 创建必须检查数据库错误和 `RowsAffected`；突破资源扣减、境界/失败次数更新、返还和调养费流水已经写入后，如果突破记录未实际落库，必须回滚整笔突破事务，避免资产和境界变化缺少追溯。

群内发送 `突破` 后，`WAITING_CONFIRM_BREAKTHROUGH` 会话允许继续在群内回复 `确认渡劫` 或 `确认代购并渡劫`。群聊只放行该突破确认步骤，其他普通会话仍按私聊流程处理。

宗门抽奖相关表：

- `SectLottery`
- `SectLotteryEntry`
- `SectLotteryPrize`
- `SectLotteryWinner`
- `SectLotteryReminder`

规则：

- 支持满人数开奖和定时开奖，两种模式互斥；定时开奖由 `StartSectLotteryScheduler` 每分钟扫描到期活动；
- 报名者必须仍为本宗门成员、本地正式账号有效、未暂停、未过期，且历史贡献不少于 `100`；历史贡献口径复用当前贡献加宗门商店已消耗贡献；
- 同一活动同一用户只能报名一次，依赖 `sect_lottery_entries(lottery_id, user_id) WHERE deleted_at IS NULL` 唯一索引；
- 报名事务只有在报名记录创建、参与人数回写和满人开奖判断全部成功提交后，才能对外返回非零参与人数、目标人数和开奖标记；事务失败或提交失败时必须返回 `0,0,false` 和错误，避免回滚后的中间人数或开奖标记触发错误回执/自动开奖。
- 开奖时会重新校验中奖候选人当前资格，不合格者跳过并继续从随机报名池中挑选后续合格成员；
- 每名用户最多中奖一次，每条卡密最多分配一次，依赖 `sect_lottery_winners(lottery_id, user_id)` 和 `sect_lottery_winners(lottery_id, prize_id)` partial unique index；
- 宗门抽奖报名和中奖幂等索引必须由启动迁移替换同名旧全量唯一索引；已执行旧 `20260618_sect_lottery_indexes` 迁移的生产库还必须通过后续补迁移重建为 `deleted_at IS NULL` 部分唯一索引，避免软删除历史报名/中奖记录阻塞新活动参与或开奖追踪；
- 宗门抽奖不扣积分、不发积分、不改变宗门资金/贡献/声望，因此不写 `PointTransaction` 或 `SectContributionLog`；创建、开奖、取消写 `AuditLog`；
- 开奖事务只写活动、中奖者和奖品分配状态；Telegram 私聊发奖在事务提交后执行，发送成功、发送失败或解密失败写回中奖和奖品状态时都必须条件更新并检查 `RowsAffected`；发送失败只把中奖和奖品标记为失败，宗主或长老可私聊重发；创建者开奖摘要必须包含中奖者名单和投递状态，但不得包含卡密明文；
- 开奖事务提交后需要私聊未中奖参与者告知本次未中奖；该通知不得放进开奖事务，发送失败只记录日志，不得回滚开奖或影响中奖卡密投递；
- 创建事务提交后会私聊提醒当前宗门成员，提醒状态写入 `SectLotteryReminder`；同一活动同一用户只能有一条提醒记录，成功提醒后补发不会重复打扰；发送失败只记录失败原因，宗主或长老可私聊 `提醒宗门抽奖 活动ID` 或 `补发宗门抽奖提醒 活动ID` 重试；提醒状态 upsert 必须检查数据库错误和 `RowsAffected`，未实际写入需记录脱敏日志，避免补发去重状态静默丢失；
- 不在群内公告中奖结果，不公开中奖卡密，避免打扰非成员和泄漏卡密。

- `AbsLiveListeningCheckpoint` 与 `live_raw_seconds` 作为旧版本兼容字段保留，但不得参与每日净修为计算。官方刷新需要把对应日的 `raw_seconds`、`official_raw_seconds`、封顶秒数和有效修为统一重写，并清零旧 `live_raw_seconds`。

## 宗门秘境

宗门秘境相关表：

- `SectSecretRealmEvent`
- `SectSecretRealmParticipant`
- `SystemConfig(sect_secret_realm_config_v1)`

迁移要求：

- `SectSecretRealmParticipant` 保存进入/结算时的总净修为、ABS 原始总听书秒数、观测原始增量、墙钟封顶、实际采用原始增量、压制后净修为、最终秘境净修为、境界、积分固定境界加成、贡献/声望倍率加成、护道加成和掉落；

秘境配置：

- 默认有普通、高阶和限时三个档位；
- 普通档位保持原 `听潮秘境` 活动口径，但积分奖励按新固定公式结算；

- 默认最低有效秘境净修为：普通 0.2 小时，高阶 0.75 小时，限时 0.2 小时；历史默认门槛普通/限时 0.2 保持为当前默认值，已保存的普通/限时旧默认 0.5 会归一到 0.2，高阶 0.3 会在配置归一化时升级到 0.75，非默认自定义值保留；
- 默认积分封顶：普通 18，高阶 30，限时 20；历史默认配置中的旧封顶值 8/14/10 会在配置归一化时升级到新默认值，非默认自定义值保留；
- 活动进行中，后台刷新、`宗门秘境`、`宗门秘境排行` 和成员进入后都会按当前 ABS 原始听书增量、墙钟封顶、秘境压制曲线和护道加成更新实时榜；实时刷新只更新预计净修为和预计奖励，不发放资产；
- 秘境结束、结算中或已结算后，`宗门秘境` 状态查询必须提示当前没有可参加的秘境；最近一次榜单仍通过 `宗门秘境排行` 查询；
- 查询当前或最近秘境时必须区分“没有记录”和数据库读取失败；`宗门秘境`、`进入宗门秘境`、`宗门秘境排行`、`宗门秘境明细` 读取失败时必须提示稍后重试并记录脱敏日志，不得把数据库错误当成暂无秘境；
- 开启秘境和进入秘境的事务内 active 秘境复核必须使用可返回错误的 checked 查询；读取失败必须回滚，不得当成无 active 秘境继续扣宗门资金或写入参与记录；
- `进入宗门秘境` 成功后的参与记录重读失败时，成功回执里的当前境界和基线净修为必须显示“读取失败”并记录脱敏日志，不得把数据库错误当成凡人或 0 小时；已提交的进入记录不得因此回滚；
- `宗门秘境排行`、实时榜和结算公告读取排行失败时必须明确显示“读取失败”或提示稍后重试，并记录脱敏日志；不得把数据库错误当成暂无参与者；
- 限时档位默认关闭，仅供运营活动启用；
- 配置保存在 `SystemConfig`，写入命令仅限超级管理员私聊执行，必须填写原因、二次确认并写 `AuditLog`；
- 开启秘境、查看秘境配置和写入秘境配置必须使用可返回错误的配置读取路径；读取失败或 JSON 解析失败时应中止并记录脱敏日志，不得按默认配置继续扣宗门资金、展示误导配置或覆盖写入；
- `SectSecretRealmMultiplierCfg.PointPercent` 仅作为历史配置兼容字段保留，当前积分不再套百分比倍率；贡献和声望继续使用 `ContributionPercent`、`PrestigePercent`；
- 开启秘境时必须把档位配置序列化保存到 `SectSecretRealmEvent.ConfigSnapshot`，结算必须按快照执行，不得因管理员后续改配置影响进行中的秘境；进入、实时刷新和结算等会影响参与资格、实时奖励估算或资产发放的路径读取快照失败时必须 fail closed，结算中遇到快照缺失或损坏必须回滚为 `active` 等待人工修复，不得回退到当前配置或默认配置继续发奖。

开启秘境会消耗宗门资金，不变更用户积分。

开启要求：

- 仅宗主或长老；
- 同一宗门同一时间只能有一个 `active` 秘境；
- 每个宗门每周最多开启 2 次秘境，普通、高阶和限时合并计算，窗口使用北京时间周一 00:00 到下周一 00:00；
- 档位必须已启用，宗门等级必须达到档位要求；
- 事件主记录依赖 `sect_secret_realm_events(realm_id) WHERE deleted_at IS NULL` 唯一索引防重复事件 ID；同一宗门同一时间只能有一个 active 秘境，依赖 `sect_secret_realm_events(sect_id) WHERE status = 'active' AND deleted_at IS NULL` 唯一索引防并发重复开启；启动迁移必须能替换同名旧全量唯一索引，避免已结束或软删除秘境阻塞新的 active 秘境开启；
- 每周次数校验必须在开启事务内、扣宗门资金前执行；事务内应先对宗门行取得写锁或等效写占位，再统计本周已创建秘境数量，避免并发同时通过周限额；
- 扣宗门资金和创建秘境事件必须在同一事务内完成；创建 `SectSecretRealmEvent` 必须检查数据库错误和 `RowsAffected`，未实际写入必须回滚宗门资金扣减，避免资金已扣但秘境不存在；
- 秘境编号、宗门名和本周开启次数等用于成功回执、实时榜或后续通知的结果，只能在扣资金、创建事件和资金消耗流水全部提交成功后发布；事务失败或提交失败时不得发送秘境已开启回执或启动实时榜；
- 写入 `SectContributionLog` 记录宗门资金消耗。

参与要求：

- 用户必须是宗门成员；
- 用户必须绑定有效 ABS 听书账号；
- 用户境界必须达到档位参与门槛；
- 同一用户同一秘境只能参与一次；
- 依赖 `sect_secret_realm_participants(realm_id, user_id) WHERE deleted_at IS NULL` 唯一索引防并发重复参与；启动迁移必须能替换同名旧全量唯一索引，避免软删除参与历史阻塞新的秘境参与记录。
- SQLite partial unique index 不能在 upsert 中指定冲突列；参与记录写入必须使用不指定冲突列的 `ON CONFLICT DO NOTHING`，并检查数据库 `Error` 与 `RowsAffected`；0 行写入只能按重复进入的幂等结果处理，不得覆盖首次进入基线。
- 进入秘境时记录 ABS 原始总听书秒数和当前总净修为作为基线，并记录当前境界；ABS 请求不得放入数据库事务。

结算要求：

- 通过 `active -> settling -> settled` 状态防重复结算；
- ABS 请求不放入数据库事务；
- 进入 `settling` 后读取参与者失败、写入参与者最终结算明细失败或发奖事务失败时，必须显式记录错误并把活动状态回滚为 `active`，避免活动卡死在 `settling` 或按空参与者误结算；
- 用户积分奖励必须使用 `applyPointDeltaInTx`；
- 积分奖励流水类型为 `sect_secret_realm_reward`；
- 单个参与者发奖必须先通过 `id + is_rewarded=false` 条件更新抢占领奖资格，再发积分、贡献、声望和掉落，避免 stale 参与记录导致重复发奖；
- 参与者最终结算明细写入和活动最终 `settled` 状态更新都必须检查数据库错误与 `RowsAffected`；未命中不得静默继续；
- 宗门秘境贡献和声望到账更新必须检查 `RowsAffected`；未命中应回滚，避免写出宗门奖励流水但资产未到账；
- 宗门贡献和声望奖励必须写 `SectContributionLog`；
- 通用宗门贡献奖励写 `SectContributionLog.BalanceAfter` 前必须重新读取更新后的 `SectMember.Contribution`，不得用发奖前的成员快照加 delta 估算余额，避免并发签到、听书贡献或 Boss 奖励导致流水余额失真；
- 单个参与者秘境净修为低于档位 `min_delta_hours` 不发奖励；
- 秘境原始听书增量必须按参与者进入时间到秘境结束时间的墙钟时长封顶，避免 ABS 异常统计或历史补录冲击奖励；
- 如果观测原始增量超过墙钟上限，必须记录明细字段并写日志，便于后续风控排查；
- 秘境压制曲线、护道者门槛和加成、贡献/声望境界倍率、奖励封顶、掉落池均来自开启时的档位快照；
- 达到门槛后，积分按固定公式结算：普通/限时为 `floor(秘境净修为小时 * 6)`，高阶为 `floor(秘境净修为小时 * 7)`，再叠加固定境界加成；普通/限时为凡人 +0、炼气 +1、筑基 +2、结丹 +3、元婴 +4、化神及以上 +5，高阶为筑基 +3、结丹 +5、元婴 +7、化神及以上 +9；积分最低为 1，并受档位积分封顶限制；
- 贡献按 `floor(秘境净修为小时 * 3)` 后套贡献境界倍率，声望按 `floor(秘境净修为小时)` 后套声望境界倍率，贡献最低为 1，并分别受档位贡献/声望封顶限制；
- 秘境掉落必须使用确定性判定，避免结算重试改变结果；炼气可掉落凝露草种子，筑基可掉落龙血果种子，结丹可掉落天心莲种子，元婴及以上可掉落秘境信物；
- 掉落写入 `Inventory` 时必须在发奖事务内使用 upsert 累加库存；
- 秘境公告、状态、排行和明细展示掉落物品名时必须使用 `inventoryItemMarkdownName`，护道者和参与者名称必须 Markdown 转义。

明细查询：

- `宗门秘境明细` 默认查看本人；
- `宗门秘境明细 用户ID` 仅限宗主、长老或超级管理员；
- 明细必须展示基线原始听书、结算原始听书、观测增量、墙钟上限、是否封顶、实际采用增量、压制后净修为、护道加成、最终秘境净修为、积分/贡献/声望和掉落。
## 个人邀请拉新

- 个人邀请链接使用 `ReferralCode` 和 `/start ref_<code>` 归因；第一版不支持自定义渠道码；邀请者必须是炼气初期及以上正式账号；创建或重新启用 `ReferralCode` 必须检查数据库错误和 `RowsAffected`，未实际写入不得向用户展示邀请链接。`ReferralCode(user_id)`、`ReferralCode(code)` 和 `ReferralActivation(invitee_id)` 的唯一索引必须只约束 `deleted_at IS NULL` 的有效行，启动迁移必须能替换同名旧全量唯一索引。
- 通过邀请链接注册的新用户为 `User.account_type=trial`，仅获得 7 天新人体验，不等价于正式邀请码开通。
- 同一邀请者每天最多激活 3 名新人体验，按北京时间自然日统计；必须通过 `ReferralDailyActivationQuota` 在创建 `ReferralActivation` 的同一事务内原子占用名额，事务失败时额度一起回滚。
- 同一 Telegram 用户只能存在一条 `ReferralActivation`，不得重复领取新人体验，也不得使用自己的邀请链接。
- 邀请注册创建新试用 `User` 和 `ReferralActivation` 时必须检查数据库错误和 `RowsAffected`；任一记录未实际写入必须回滚每日激活额度和本地试用档案，`ReferralActivation` 唯一冲突继续映射为已尝试。
- 试用用户体验期内累计听书满 10 小时后，发送 `新人任务` 可领取 7 天新人体验延期；该延期直接更新当前试用账号 `expire_at`，不生成 `RenewCode`，不能转赠、囤积或给正式用户使用。任务累计秒数必须按试用开始到当前/结束的实际墙钟时长封顶，避免 ABS 日聚合把试用开始前的当日历史听书计入奖励条件。
- 邀请者每产生 1 名有效新人奖励 10 积分，流水类型为 `referral_reward`，每月最多 150 积分；月奖励额度必须通过 `ReferralMonthlyRewardQuota` 原子占用，超过月上限时新人仍可领取体验延期，但邀请者不再得分。
- 新人听书任务以 `daily_listening_stats.capped_seconds` 汇总试用窗口内 ABS 原始听书时长，单日最多按 24 小时计入。
- 试用用户不能使用普通续期卡；普通续期卡核销前必须拒绝 `account_type=trial`，且不得把卡密标记为已使用。
- 试用用户使用正式邀请码时走转正流程：核销 `InviteCode`，将 `User.account_type` 改为 `formal`，之后才可使用普通续期卡。
- 个人邀请链接、试用转正有效期、新人任务延期和邀请者奖励等返回给调用方的事务结果，只能在事务提交成功后发布；事务失败或提交失败时必须清空链接、有效期、奖励积分和发奖标记等结果，任务未完成路径可保留已读取的听书进度秒数。
- 邀请注册本地写入失败时，必须回滚刚创建的 ABS 账号；若 ABS 回滚失败，不得提交本地归因或试用账号状态，并提示管理员核查。
